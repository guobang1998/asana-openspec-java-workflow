# Asana OpenSpec Java Workflow 团队介绍

## 一句话

这是一套给 Java/Spring/MyBatis 后端团队用的 AI 交付流程：需求先进 Asana，确认成 PRD，再进入 OpenSpec 变更记录，最后由 Codex 按规范实现、测试、Review、开 PR。

## 为什么要用

我们不希望 AI 直接看一段聊天就改代码。新流程要求每个需求都有：

- Asana 来源。
- PRD 需求文档。
- OpenSpec 变更文档。
- CodeGraph 影响面定位。
- Java/Spring/MyBatis 编码规范。
- MySQL 数据库安全规则。
- 单测/集成/手动验证记录。
- PR 前质量门禁。

## 工具分工

| 工具 | 职责 |
|---|---|
| Asana | 需求入口、负责人、优先级、状态流转 |
| PRD | 业务背景、目标、边界、验收标准 |
| OpenSpec | 变更账本：proposal、design、tasks、specs |
| CodeGraph | 代码入口、调用链、影响面定位 |
| Codex Skills | 执行流程、规范、Review、测试门禁 |
| MySQL MCP | 查结构、查数据、验证 SQL、受控写入 |
| Git PR | 代码 Review、CI、合并 |

## 标准流程

可视化版本见 `流程图.md`。

```text
Asana 新需求
-> 生成 PRD
-> 人工确认 PRD
-> 创建 OpenSpec change
-> CodeGraph 定位影响面
-> Plan Review
-> Codex 实现
-> 单测/集成/手动验证
-> Java Review / Security Review / MySQL Guard
-> PR Quality Gate
-> 开 PR
-> 合并
-> OpenSpec archive
-> Asana 完成
```

大重构不走单个 OpenSpec change。大重构走：

```text
Asana Epic
-> CodeGraph Discovery
-> 重构 RFC
-> 测试基线
-> Phase 拆分
-> 多个 OpenSpec changes
-> 小 PR 串行合并
-> 清理旧代码
-> 复盘/archive
```

## 团队规则

- 需求不清楚，不写代码。
- PRD 未确认，不进入实现。
- 实现偏差时，先改 PRD/OpenSpec，再改代码。
- 每个代码改动必须对应 PRD、OpenSpec 或 `tasks.md`。
- 行为变更必须有测试，或说明不能自动化原因。
- 涉及 DB 必须写影响面、SQL、回滚方案。
- `DELETE`、`CREATE`、`ALTER` 必须确认。
- PR 前必须列出验证结果和未覆盖项。
- 大重构必须先写 RFC，并拆多个 OpenSpec changes。

## 适合先试点的需求

- 0.5 到 2 天能完成。
- 影响面不超过 2 到 3 个模块。
- 有明确验收标准。
- 最好是 Java 后端功能、接口优化、Bugfix、小型配置或 SQL 改动。

## 不适合第一轮试点的需求

- 大重构。
- 多系统联动。
- 生产高风险数据修复。
- 权限模型大改。
- 没有验收标准的模糊需求。

## 团队成员怎么配合

产品/需求方：

- 在 Asana 写清背景、目标、验收标准。
- 确认 PRD。
- 验收最终结果。

开发 Owner：

- 确认 OpenSpec design/tasks。
- 决定风险和取舍。
- Review PR。

Codex：

- 整理 PRD。
- 生成 OpenSpec change。
- 定位代码影响面。
- 实现代码。
- 生成测试和 Review 清单。

Reviewer：

- 看需求是否跑偏。
- 看关键代码、事务、权限、SQL、安全。
- 确认测试证据。
