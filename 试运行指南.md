# 试运行指南

## 目标

用两条真实小需求跑通完整流程：一条已有 Asana 的明确需求，一条没有 Asana 的模糊需求。不要一开始追求自动化，先验证流程是否顺。

## 准备

1. 选择一个 Java/Spring/MyBatis 仓库。
2. 把 `assets/templates/AGENTS模板.md` 内容合并到项目 `AGENTS.md`。
3. 按 `安装配置指南.md` 安装 OpenSpec。
4. 在仓库初始化 OpenSpec。
5. 如果要减少 AI 乱定位，按 `安装配置指南.md` 安装并初始化 CodeGraph。
6. 如果要连 MySQL MCP，准备专门 AI 账号，参考 `assets/templates/MySQL数据库安全模板.md`。
7. 确认 Codex 已加载本插件 skills。源码 `.codex-plugin/plugin.json` 的版本必须与 `codex plugin list` 显示版本一致；新开会话的可用 skills 必须同时包含 `asana-openspec-java-workflow:setup-workflow`、`asana-openspec-java-workflow:wayfinder-workflow` 和 `asana-openspec-java-workflow:asana-openspec-delivery`。任一项缺失时，重装插件并重启 Codex 或新开目标仓库会话；项目 `AGENTS.md` 只能兜底已有规则，不能证明新增 skill 已加载。

## 选择试点需求

推荐：

- 小 Bugfix。
- 小接口新增。
- 小字段新增。
- 小查询条件优化。
- 小配置/日志改动。

避免：

- 大范围重构。
- 多服务联动。
- 生产数据修复。
- 完全没有背景、目标、业务场景或验收方向的空泛需求。
- 有探索价值但验收标准未成形的模糊需求，不按普通小需求试点；可以作为路径 B 试跑，先澄清再进入 PRD，不能直接实现。

如果试点需求实际是大重构，不要继续套小需求流程。改用：

```text
large-refactor-workflow
assets/templates/重构RFC模板.md
assets/templates/重构检查清单.md
docs/workflow/大需求分段审核与高风险复查协议.md
```

## 第一次试跑路径

### 路径 A：已有 Asana 的明确小需求

目标：验证标准交付链路。

#### 1. Asana 新建需求

任务描述至少包含：

```md
## 背景

## 目标

## 验收标准

## 附件/截图/链接
```

#### 2. 让 Codex 生成 PRD

提示词：

```text
请按 asana-openspec-java-workflow 的 prd-writer，把这个已有 Asana 的明确小需求整理成 PRD md。
如果信息不够，先列待确认问题，不要进入实现。
```

输出位置建议：

```text
docs/prd/<需求名>.prd.md
```

#### 3. 人工确认 PRD

检查：

- 背景是否准确。
- 目标是否明确。
- 不做什么是否写清。
- 验收标准是否可测试。
- 是否有待确认问题。

#### 4. 创建 OpenSpec change

提示词：

```text
PRD 已确认。请创建 OpenSpec change，生成 proposal.md、design.md、tasks.md、specs，并按 Java/Spring/MyBatis 后端流程拆解。
```

命名建议：

```text
add-xxx
fix-xxx
optimize-xxx
```

#### 5. CodeGraph 定位影响面

提示词：

```text
如果项目存在 .codegraph/，请先按 codegraph-context-guard 定位入口、调用链、影响面，再更新 OpenSpec design/tasks。
```

输出要有：

```text
入口：
调用链：
影响文件：
风险点：
需要验证：
```

#### 6. 实现前确认纪律

提示词：

```text
如果当前会话能使用 asana-openspec-java-workflow:coding-discipline skill，请按该 skill 确认假设、范围、不做内容、验证方式，再开始实现。
如果当前会话没有暴露 asana-openspec-java-workflow:coding-discipline skill，请按项目 AGENTS.md 的“AI 写代码纪律”执行：先确认假设、范围、不做内容、验证方式，再开始实现。
日志和注释使用中文。
```

期望输出：

```text
假设：
范围：
不做：
验证：
风险：
```

说明：

- `coding-discipline` 是本插件随包提供的 skill；在 Codex 可用 skills 清单里通常显示为 `asana-openspec-java-workflow:coding-discipline`。
- 只有插件被 Codex 正确加载后，才能直接按 skill 名称调用。
- 如果 skill 名称不可用，`AGENTS.md` 中的“AI 写代码纪律”就是兜底规则，试跑可以继续。

#### 7. Codex 实现

要求：

- 按 `tasks.md` 做。
- 小步修改。
- 不做无关重构。
- 偏差先回 OpenSpec。
- 涉及 Mapper/SQL、列表、搜索、统计、导出、分页或排序时，先按 `sql-performance-review` 编码模式设计查询形状、分页排序、索引和证据，再写代码。

#### 8. 测试

提示词：

```text
请按 java-test-strategy 判断需要哪些单测、集成测试和手动验证，并补齐必要测试。
```

#### 9. Review 和门禁

提示词：

```text
请依次执行：
1. codegraph-context-guard impact 复查
2. 填写 PR评审清单.md 的分段审核基础判断
3. 如果命中 B 类、A 类或大重构，补充扩展字段，并判断是否需要高风险复查
4. java-backend-review
5. springboot-security-review，如果涉及接口/权限/输入/敏感数据
6. sql-performance-review 审查模式，如果涉及 Mapper/SQL、列表、搜索、统计、导出、分页或排序
7. mysql-db-guard，如果涉及数据库
8. pr-quality-gate
```

#### 10. 开 PR

PR 描述必须包含：

```md
## 需求来源
来源类型：Asana
Asana:
OpenSpec change-id:

## 变更内容

## 验证结果
- 编译：
- 单测：
- 集成测试：
- 手动验证：
- 未覆盖项：

## 风险

## 回滚方案

## 分段审核与高风险复查
风险等级: A 类 / B 类 / C 类
命中条件:
是否建议高风险复查:
复查类型:
未触发原因:
复查方式:
复查记录:
复查结论:
B/A 类扩展字段:
A 类强制字段:
负责人确认人:
负责人确认时间:
负责人确认依据:

## Review 关注点
```

#### 11. 合并后归档

合并后：

```text
OpenSpec verify
OpenSpec archive
有 Asana 时标记完成
```

路径 A 验收：

- PRD 关联 Asana。
- OpenSpec change 完整。
- PR Gate 有测试和风险说明。

### 路径 B：没有 Asana 的模糊需求

目标：验证多入口和 brainstorming 链路。

#### 1. 用户输入一个模糊需求

示例提示词：

```text
这不是 Asana 任务，是会议里提到的想法。现在需求还模糊，请按多入口需求路由处理；需要时用 brainstorming 澄清，不要写代码。
```

#### 2. AI 判断入口和清晰度

期望：

- 标记 `需求来源：用户对话 / 会议纪要`。
- 标记 `是否已有 Asana：否 / 待补`。
- 判断为模糊需求时，不进入实现。

#### 3. brainstorming 或轻量澄清

小范围、低风险、目标基本可判断时，可先做轻量澄清。

命中新功能探索、方案分歧较大、用户体验/业务口径不清、范围不清或可能重构时，使用 `superpowers:brainstorming`，输出：

```text
目标：
范围：
不做什么：
候选方案：
待确认问题：
```

#### 4. 生成 PRD 草稿

PRD 必须记录：

```text
需求来源：用户对话 / 会议纪要
是否已有 Asana：否 / 待补
跟踪要求：无需跟踪 / PR 前补 / 排期前补 / 跨团队前补
brainstorming 记录：
待补跟踪项：
```

#### 5. 用户确认 PRD

确认后再创建 OpenSpec change。需要团队追踪、排期或跨团队协作时，后补 Asana。

#### 6. 继续标准交付

```text
OpenSpec change
-> CodeGraph impact
-> 实现
-> 测试
-> PR Gate
```

路径 B 验收：

- brainstorming 结果被 PRD 吸收。
- 模糊需求没有直接进入实现。
- PR Gate 能追踪需求来源和未补 Asana 状态。

## 成功标准

- 需求 md 已生成并确认。
- PRD 已记录需求来源、是否已有 Asana 和跟踪要求。
- OpenSpec change 已生成。
- 实现能追溯到 `tasks.md`。
- 测试和 Review 有记录。
- PR 描述完整。
- 涉及 SQL/查询接口时，PR Gate 有 SQL 性能结论、分页/排序说明和 `EXPLAIN` 或低风险理由。
- 合并后 OpenSpec 已归档。

## 第一次试跑复盘

跑完记录：

- 哪一步卡住。
- 哪个模板缺字段。
- 哪个 skill 触发不自然。
- 哪些规则太严格或太松。
- 是否需要自动化 Asana 状态同步。
