# 试运行指南

## 目标

用一条真实小需求跑通完整流程。不要一开始追求自动化，先验证流程是否顺。

## 准备

1. 选择一个 Java/Spring/MyBatis 仓库。
2. 把 `assets/templates/AGENTS模板.md` 内容合并到项目 `AGENTS.md`。
3. 按 `安装配置指南.md` 安装 OpenSpec。
4. 在仓库初始化 OpenSpec。
5. 如果要减少 AI 乱定位，按 `安装配置指南.md` 安装并初始化 CodeGraph。
6. 如果要连 MySQL MCP，准备专门 AI 账号，参考 `assets/templates/MySQL数据库安全模板.md`。

## 选择试点需求

推荐：

- 小 Bugfix。
- 小接口新增。
- 小字段新增。
- 小查询条件优化。
- 小配置/日志改动。

避免：

- 大范围重构。
- 多服务联动。
- 生产数据修复。
- 无验收标准需求。

## 第一次试跑步骤

### 1. Asana 新建需求

任务描述至少包含：

```md
## 背景

## 目标

## 验收标准

## 附件/截图/链接
```

### 2. 让 Codex 生成 PRD

提示词：

```text
请按 asana-openspec-java-workflow 的 prd-writer，把这个 Asana 需求整理成 PRD md。
如果信息不够，先列待确认问题，不要进入实现。
```

输出位置建议：

```text
docs/prd/<需求名>.prd.md
```

### 3. 人工确认 PRD

检查：

- 背景是否准确。
- 目标是否明确。
- 不做什么是否写清。
- 验收标准是否可测试。
- 是否有待确认问题。

### 4. 创建 OpenSpec change

提示词：

```text
PRD 已确认。请创建 OpenSpec change，生成 proposal.md、design.md、tasks.md、specs，并按 Java/Spring/MyBatis 后端流程拆解。
```

命名建议：

```text
add-xxx
fix-xxx
optimize-xxx
```

### 5. CodeGraph 定位影响面

提示词：

```text
如果项目存在 .codegraph/，请先按 codegraph-context-guard 定位入口、调用链、影响面，再更新 OpenSpec design/tasks。
```

输出要有：

```text
入口：
调用链：
影响文件：
风险点：
需要验证：
```

### 6. 实现前确认纪律

提示词：

```text
请按 coding-discipline 确认假设、范围、不做内容、验证方式，再开始实现。
```

### 7. Codex 实现

要求：

- 按 `tasks.md` 做。
- 小步修改。
- 不做无关重构。
- 偏差先回 OpenSpec。

### 8. 测试

提示词：

```text
请按 java-test-strategy 判断需要哪些单测、集成测试和手动验证，并补齐必要测试。
```

### 9. Review 和门禁

提示词：

```text
请依次执行：
1. codegraph-context-guard impact 复查
2. java-backend-review
3. springboot-security-review，如果涉及接口/权限/输入/敏感数据
4. mysql-db-guard，如果涉及数据库
5. pr-quality-gate
```

### 10. 开 PR

PR 描述必须包含：

```md
## 需求来源
Asana:
OpenSpec change-id:

## 变更内容

## 验证结果
- 编译：
- 单测：
- 集成测试：
- 手动验证：
- 未覆盖项：

## 风险

## 回滚方案

## Review 关注点
```

### 11. 合并后归档

合并后：

```text
OpenSpec verify
OpenSpec archive
Asana 标记完成
```

## 成功标准

- 需求 md 已生成并确认。
- OpenSpec change 已生成。
- 实现能追溯到 `tasks.md`。
- 测试和 Review 有记录。
- PR 描述完整。
- 合并后 OpenSpec 已归档。

## 第一次试跑复盘

跑完记录：

- 哪一步卡住。
- 哪个模板缺字段。
- 哪个 skill 触发不自然。
- 哪些规则太严格或太松。
- 是否需要自动化 Asana 状态同步。
