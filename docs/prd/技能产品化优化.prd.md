# PRD：Asana OpenSpec Java Workflow 技能产品化优化

## 背景

当前 `asana-openspec-java-workflow` 已经具备比较完整的 Java/Spring/MyBatis 交付治理能力，覆盖 PRD、OpenSpec、CodeGraph、Java Review、SQL 性能、MySQL 安全、测试策略、PR Quality Gate 和大重构分阶段流程。

对比 `mattpocock/skills` 后，差距不在 Java 后端专业规则，而在 skill 产品化程度、安装/初始化体验、领域上下文沉淀、TDD/Debug 反馈回路、架构体检和任务切片方式。尤其是 `wayfinder` 体验差距明显：Matt 的用户可以拿一个“很大、很模糊、一个会话装不下”的想法，让 agent 先画出共享地图、前沿票据、阻塞关系和待澄清区域；而本仓库目前更偏“用户已经知道要用 PRD/OpenSpec/Review 哪个节点”，缺少一个替用户找路的入口。

Matt 的做法更像“可安装、可初始化、可组合、可自我维护的技能产品”；本仓库更像“强门禁、强流程、强后端规范的团队交付工具箱”。

本次优化目标是吸收 Matt skills 的产品化和执行稳定性优点，同时保留本仓库的 Java/SQL/OpenSpec/DB 风险控制强项。

## 目标

1. 降低团队成员首次接入成本，让新 Java 仓库能通过一个 setup skill 完成环境检查、模板选择和接入建议。
2. 提升 skill 触发稳定性，明确区分用户显式调用、模型自动触发和门禁型 skill。
3. 增加轻量领域上下文层，让 PRD/OpenSpec 之外的领域词汇、核心 ID、业务边界和 ADR 能跨会话复用。
4. 强化 TDD/Debug 反馈回路，让 bugfix 和行为变更先有可失败的验证入口，再改实现。
5. 新增 `wayfinder-workflow` 能力，让用户输入模糊大目标后，agent 先产出共享地图、当前前沿、阻塞关系、待明确区域和出界范围，而不是要求用户自己知道下一步该调用哪个 skill。
6. 增加任务切片机制，把大需求拆成一个上下文窗口内可完成、可验证、可回滚的 frontier ticket。
7. 优化 Review 输出结构，把规范问题、需求偏差、风险门禁和测试缺口分轴呈现，减少一个大门禁里混杂所有结论。
8. 保持既有 Java/Spring/MyBatis、SQL 性能、MySQL 安全、OpenSpec 和 PR Gate 规则不降级。

## 不做什么

1. 不替换 OpenSpec，不把 PRD、OpenSpec、tasks 全部改成 Matt 风格 tickets。
2. 不移除现有 `sql-performance-review`、`mysql-db-guard`、`pr-quality-gate` 等后端门禁。
3. 不默认自动修改 Asana、GitHub PR、生产数据库或远程状态。
4. 不第一阶段做完整跨 agent 编排平台。
5. 不复制 Matt 的通用 TypeScript/前端语境，只吸收可迁移的方法。
6. 不把所有 skill 改成更短导致安全、DB、生产发布语义不清。

## 术语表

| 术语 | 说明 |
|---|---|
| setup skill | 用户显式调用的初始化 skill，用来检查目标仓库并生成接入建议或模板文件。 |
| skill 产品化 | skill 不只是一段规则，还包含安装、初始化、版本、校验、示例和升级路径。 |
| 领域上下文 | 跨会话复用的业务术语、核心实体、核心 ID、边界、外部系统信任源和常见 ADR。 |
| ADR | Architecture Decision Record，记录重要架构决策、取舍、时间和后续影响。 |
| seam | 可测试边界，例如 Service 方法、Mapper 查询、Controller API、状态机动作或消息处理入口。 |
| red-capable 反馈回路 | 修改前能失败、修改后能通过的自动化或半自动化验证入口。 |
| wayfinding map | 面向模糊大目标的共享路线图，记录目标、已决策、待明确、出界范围和子票据。 |
| destination | 本次找路最终要抵达的产物或决策，例如 PRD、OpenSpec 方案、重构 RFC、迁移路线或可执行任务拆分。 |
| frontier | 当前已明确、未阻塞、可被一个会话处理的下一批票据。 |
| fog of war | 已知还会影响目标、但当前问题还不能精确定义的待明确区域。 |
| frontier ticket | 当前上下文窗口内可独立完成的小任务，包含目标、边界、依赖、验收和阻塞关系。 |
| Review 双轴 | 至少区分 Standards Review 和 Spec Review，分别看代码规范/工程风险与需求符合度。 |

## 用户/系统交互

### 角色

| 角色 | 说明 |
|---|---|
| Workflow 维护者 | 维护本插件、模板、skill、版本和试运行记录。 |
| Java 业务开发者 | 在目标业务仓库中使用本插件推进需求、修复、重构和 PR。 |
| Codex Agent | 按插件 skill 执行 PRD、OpenSpec、代码修改、测试和 Review。 |
| 可选 Reviewer | Claude Code、人工 Reviewer 或其他外部审查者，只作为补充审查输入。 |

### 主流程

1. 用户安装或更新插件。
2. 用户在目标 Java 仓库显式调用 setup skill。
3. setup skill 检查仓库是否存在 `AGENTS.md`、OpenSpec、`.codegraph/`、PRD 目录、测试命令、Maven/Gradle、MySQL MCP 说明和现有领域文档。
4. setup skill 输出接入结论，并在用户确认后生成或更新模板文件。
5. 后续需求仍按 `prd-writer` -> OpenSpec -> CodeGraph -> 实现 -> 测试 -> Review -> PR Gate 推进。
6. 需求推进过程中，领域词汇和 ADR 进入轻量上下文层；行为变更进入 TDD/Debug 反馈回路；复杂实现进入 frontier ticket 切片。
7. PR 前门禁保留，但输出按规范、需求、测试、SQL/DB、安全、风险分轴展示。

### Wayfinder 用户体验流程

当用户输入“我们要优化这个 workflow”“这个模块要治理一下”“参考某套技能体系升级一下”这类大而模糊的目标时，agent 不要求用户先选择 `prd-writer`、`large-refactor-workflow` 或 `java-test-strategy`，而是进入 `wayfinder-workflow`：

```text
模糊大目标
-> 命名 destination
-> 广度澄清当前未知
-> 创建 wayfinding map
-> 生成第一批 frontier tickets
-> 标记 blocking / not yet specified / out of scope
-> 停止，不在同一会话里同时执行票据
```

用户后续可以说“继续这个 map”，agent 加载 map 的低分辨率视图，自动选择第一个未阻塞、未认领的 frontier ticket，解决一个决策后关闭票据，并把决策指针追加回 map。

## 业务规则

### BR-1 Setup 必须幂等

同一个目标仓库多次运行 setup skill，不能重复追加相同规则、重复创建同名模板或覆盖用户已有内容。需要修改既有文件时，必须先展示计划和影响范围。

### BR-2 Skill 必须标明触发类型

每个 skill 在文档中必须标明：

| 触发类型 | 说明 |
|---|---|
| user-invoked | 用户显式点名或命令式调用。 |
| model-invoked | 命中任务条件时模型主动使用。 |
| gate | Review、PR、DB、发布等门禁节点使用。 |

### BR-3 保留 Java 后端硬门禁

涉及 Mapper/SQL、列表、搜索、统计、导出、分页、排序、动态查询、DB 写入、DDL、权限、状态机、事务边界时，仍必须按现有专项 skill 执行，不允许被轻量流程绕过。

### BR-4 领域上下文不能替代 PRD/OpenSpec

领域上下文只记录稳定事实、术语和决策背景。业务验收仍以 PRD 为准，行为变更仍以 OpenSpec 为准。冲突时必须回到 PRD/OpenSpec 更新。

### BR-5 Bugfix 和行为变更必须先找 seam

除纯文档、注释、格式化和配置说明外，bugfix 与行为变更必须先说明测试 seam、失败方式和验证命令。无法自动化时，要给出手动验证脚本或 SQL/接口证据。

### BR-6 Frontier ticket 服务 OpenSpec tasks

frontier ticket 是执行切片，不替代 OpenSpec `tasks.md`。OpenSpec 负责变更账本，frontier ticket 负责当前上下文窗口内怎么安全完成。

### BR-7 Wayfinder 必须先找路，不直接做

`wayfinder-workflow` 默认只做规划和决策清晰化，不直接实现目标。一次会话最多只做一件事：

1. 画 map。
2. 或解决一个 frontier ticket。

如果用户明确要求执行票据内的任务，必须在 ticket 的 Notes 中写明本 ticket 允许执行到什么程度，并仍然遵守 PRD/OpenSpec/Java/SQL/DB 门禁。

### BR-8 Map 是索引，不是正文仓库

wayfinding map 只保存低分辨率索引：

- destination。
- notes。
- decisions so far。
- not yet specified。
- out of scope。
- child tickets 链接。

具体问题、调研过程、原型、人工确认和结论只保存在对应 ticket 或关联资产中，避免 map 变成大杂烩。

### BR-9 票据必须可由一个会话处理

每个 frontier ticket 必须聚焦一个问题或一个解阻塞动作。票据太大时必须继续拆分；票据太虚时放回 `not yet specified`，不要假装已经可执行。

### BR-10 阻塞关系要可视化

优先使用 issue tracker 的原生父子关系、阻塞关系、标签和 assignee 表达 map、child tickets、claim 和 frontier。没有 tracker 时，降级到本地 markdown 文件协议，但字段必须保持一致。

### BR-11 票据类型必须区分 HITL / AFK

票据类型至少包含：

| 类型 | HITL / AFK | 说明 |
|---|---|---|
| research | AFK | 阅读文档、第三方 API、本地知识库，产出摘要资产。 |
| prototype | HITL | 生成粗糙原型、轮廓或示例，让用户反馈行为或形态。 |
| grilling | HITL | 一次一个问题澄清业务、领域模型、取舍和验收。 |
| task | HITL 或 AFK | 为了做出决策而必须先完成的解阻塞动作。 |

### BR-12 Review 输出必须分轴

Review 结论至少拆为：

1. Standards：编码规范、结构、可维护性、事务、异常、日志。
2. Spec：是否符合 PRD/OpenSpec 验收标准。
3. Tests：单测、集成、手动验证、未覆盖项。
4. Risk Gates：SQL/DB、安全、性能、回滚、上线观察。

### BR-13 发布必须有版本和校验记录

新增或修改 skill 后，必须更新插件版本或变更记录，并执行结构校验、文档链接校验和关键 skill 示例校验。

## 日志/监控

本需求不涉及线上日志。

插件侧需要记录以下执行证据：

1. setup skill 输出仓库检查结果、用户确认项和生成/修改文件列表。
2. skill 质量审查输出问题清单和修复结论。
3. wayfinder map 记录 destination、decisions so far、not yet specified、out of scope 和 child tickets。
4. 票据记录 claim 人、票据类型、阻塞关系、解决结论、关联资产和关闭时间。
5. 试运行记录记录真实需求中的触发准确率、重复规则、遗漏规则和用户手动纠正点。

## 权限/安全

1. setup skill 不能默认写入外部系统状态。
2. setup skill 修改文件前必须展示目标路径和变更类型。
3. 涉及 `AGENTS.md`、MySQL MCP、数据库账号、密钥、生产地址时，不能输出真实敏感值。
4. 新增领域上下文时，必须允许标记“内部事实”“不可外发事实”“可进入 PRD/OpenSpec 的事实”。
5. wayfinder ticket claim 必须可见，避免多个 agent 重复处理同一个票据。
6. 任何 DB 相关规则仍由 `mysql-db-guard` 控制。

## 验收标准

### AC-1 Setup 接入

在一个未接入本插件的 Java/Spring/MyBatis 测试仓库中调用 setup skill，必须输出：

- 是否存在 `AGENTS.md`。
- 是否存在 OpenSpec。
- 是否存在 `.codegraph/`。
- 构建工具类型：Maven / Gradle / 未识别。
- 测试命令候选。
- PRD/OpenSpec/领域上下文建议路径。
- MySQL MCP 是否需要配置。
- 需要用户确认的写入计划。

### AC-2 幂等验证

同一测试仓库连续运行 setup skill 两次，不得产生重复段落、重复模板或无提示覆盖。

### AC-3 既有交付链路不回退

已有流程仍能按以下链路执行：

```text
prd-writer -> OpenSpec -> CodeGraph -> coding-discipline -> Java/SQL 实现 -> java-test-strategy -> Review -> pr-quality-gate
```

### AC-4 Wayfinder 找路体验

给出一个模糊大目标，例如“参考 Matt Pocock skills 优化本插件体验”，调用 `wayfinder-workflow` 后必须产出：

- destination。
- notes。
- decisions so far 初始为空或只包含已确认前提。
- not yet specified。
- out of scope。
- 第一批 frontier tickets。
- 每个 ticket 的类型：research / prototype / grilling / task。
- 每个 ticket 的 HITL / AFK 属性。
- blocking 关系。

同一会话只允许创建 map，不允许同时解决第一个 ticket。

### AC-5 Wayfinder 继续推进

给定一个已存在 map，agent 必须：

- 只加载 map 低分辨率视图。
- 为确定 frontier，可额外读取 `## Frontier` 指向 ticket 及其 blocker 链的 YAML frontmatter；不得读取无关 ticket 或 ticket 正文。
- 自动选择第一个未阻塞、未认领的 frontier ticket，除非用户指定票据。
- 先 claim，再处理。
- 一次只解决一个 ticket。
- 关闭 ticket 后，把决策指针追加到 map 的 decisions so far。
- 将新暴露的问题变成新 ticket，或放入 not yet specified。
- 将出界内容写入 out of scope，而不是混入 decisions so far。

### AC-6 SQL 性能三模式不回退

`sql-performance-review` 必须继续保留：

- 编码模式。
- 审查模式。
- 用户显式调用时的 SQL-only 专项模式。

### AC-7 TDD/Debug 规则可触发

对一个明确 bugfix 任务，agent 必须先输出：

- seam。
- 当前失败现象。
- 复现方式或 red-capable 验证入口。
- 修复后验证命令。

没有自动化测试条件时，必须说明手动验证证据。

### AC-8 Review 分轴输出

对一次 Java/Spring/MyBatis 改动执行 Review，输出必须分为：

- Standards。
- Spec。
- Tests。
- Risk Gates。

涉及 SQL 时必须包含 SQL 性能结论；涉及 DB 写入/DDL 时必须引用 `mysql-db-guard`。

### AC-9 Frontier ticket 可执行

对一个复杂 OpenSpec task，frontier ticket 必须包含：

- 当前切片目标。
- 不做什么。
- 输入依赖。
- 输出文件或接口。
- 验收标准。
- 阻塞关系。
- 回滚或放弃方式。

### AC-10 试运行通过

至少用 4 条真实或仿真需求试运行：

1. 一个模糊大目标，用 `wayfinder-workflow` 画 map。
2. 明确小 bugfix。
3. 涉及 Mapper/SQL 的查询优化。
4. 一个需要拆切片的中等复杂需求。

每条都要记录触发是否准确、是否减少重复确认、是否保留原有门禁。

## 风险与回滚

| 风险 | 影响 | 缓解 |
|---|---|---|
| 新 skill 过多导致触发混乱 | agent 误用、重复输出 | 明确触发类型，新增 `skill-quality-review`。 |
| Wayfinder 和 OpenSpec 边界不清 | map、PRD、OpenSpec 三套文档打架 | map 只负责找路和决策指针；PRD 负责业务合同；OpenSpec 负责行为变更账本。 |
| Map 变成长篇正文 | 用户看不清当前 frontier | 规定 map 是索引，细节放 ticket 和资产。 |
| Agent 一次解决多个 ticket | 上下文漂移、并发冲突 | 明确一次会话最多处理一个 ticket。 |
| setup 自动写入破坏用户文件 | 用户已有规则被覆盖 | 默认先展示计划，用户确认后写入，写入幂等。 |
| 领域上下文和 PRD/OpenSpec 冲突 | 需求事实源混乱 | 规定 PRD/OpenSpec 优先，领域上下文只记录稳定事实。 |
| TDD 规则让小改变重 | 小修复成本上升 | 允许纯文档/注释/格式化跳过；无自动化时给手动证据。 |
| Review 分轴后输出变长 | 用户阅读成本上升 | PR Gate 汇总只展示结论和缺口，详情折叠到对应 skill。 |

回滚策略：

1. 新增 skill 第一阶段只作为可选能力，不设为硬门禁。
