# 交付收敛编排设计

## 文档信息

| 项目 | 内容 |
|---|---|
| 状态 | 已确认设计，待实现计划 |
| 日期 | 2026-07-20 |
| 适用范围 | `asana-openspec-java-workflow` 插件及其 Java/Spring/MyBatis 交付流程 |
| 参考设计 | [`lcw363/subagent-delivery`](https://github.com/lcw363/subagent-delivery) 的 fresh Reviewer、固定 checkpoint、修复后复审机制 |

## 背景

当前工作流已有 PRD Review、Plan Review、Java 后端评审、SQL 性能评审、安全评审、测试策略和 PR Quality Gate。各门禁能发现问题，但如果每个检查面独立返回、每轮修复后重新全量审查，用户会持续看到“发现问题—修复—又发现问题”的过程。

问题不在于发现问题太多，而在于：

- 首轮审查缺少统一输入合同和完整检查矩阵，容易挤牙膏式发现问题。
- 不同专业 Skill 分别向用户汇报，缺少问题去重、归因和集中修复。
- Writer 只修单点，没有强制扫描同类位置、上下游影响和回归测试。
- 修复后重新全量盲审，重复成本高，也容易扩大审查范围。
- PR Gate 承担了过多普通缺陷发现职责，而不是最终证据验收职责。
- 内部质量过程直接暴露给用户，用户被迫参与技术修复调度。

## 目标

新增交付收敛编排层，在不削弱现有专业门禁的前提下，实现：

1. 计划阶段一次完整预检，先消除结构性缺口再开发。
2. 代码阶段一次综合深审，形成完整、去重的问题账本。
3. 按根因集中修复，并在修复过程中扫描同类问题和上下游影响。
4. 后续只做增量复审，不机械重复全量审查。
5. 普通问题在内部收敛，用户只处理业务决策、范围变化和剩余高风险。
6. 普通任务目标为一个修复包完成收敛，平均内部 fix loop 不超过一次。

## 非目标

- 不降低 P0-P3 成立问题的修复标准。
- 不用“限制 Reviewer 报问题数量”制造虚假通过。
- 不让编排层替代 PRD、OpenSpec、CodeGraph、测试或专业 Review Skill。
- 不让 Reviewer 接手生产代码修改。
- 不强制所有任务使用多个 Reviewer、worktree 或自动 checkpoint commit。
- 不把无关历史问题带入当前需求，造成范围无限扩张。
- 不自动执行 push、PR、部署、生产操作或数据库变更。

## 核心原则

### 1. 审查是内部交付能力

内部可以存在 Review、修复和复审，但用户默认只看到范围确认、必要决策和最终结果。能依据已确认规格自行修复的问题，不要求用户逐轮确认。

### 2. 首轮看全，后续看差异

首轮 Reviewer 必须完成固定检查矩阵，并明确未覆盖项。修复后的 Reviewer 只检查问题关闭情况、新增 diff、受影响调用链和回归证据。

### 3. 按根因修复，不按评论逐条改

多个表现属于同一根因时合并成一个修复包。Writer 修复根因时同步扫描同类代码、上下游影响和测试缺口。

### 4. 风险决定审查强度

C 类小改使用轻量路径；普通需求使用一次综合 Review 和一次增量复审；A/B 类高风险需求才增加专项 Reviewer 或额外复审。

### 5. 结论绑定固定版本

计划结论绑定 PRD/OpenSpec/tasks 内容 hash；代码结论绑定 Git SHA。无法提交时绑定 task-only diff hash。目标版本变化后旧结论只作为历史证据，不直接作为新版本通过依据。

## 总体架构

新增编排 Skill，暂定名 `delivery-convergence-loop`。它不直接判断 Java、SQL、安全或测试细节，只负责固定输入、调度专业 Skill、维护问题账本、控制修复和复审、管理用户可见性。

```text
已确认 PRD / OpenSpec / tasks
        ↓
PLAN_PREFLIGHT
计划一次性预检
        ↓
DEV_ACTIVE
开发、测试、自检
        ↓
COMPREHENSIVE_REVIEW
合并规格、Java、SQL、安全、测试检查面
        ↓
FIXING
集中修复、同类扫描、回归测试
        ↓
INCREMENTAL_REVIEW
只复审修复点、新 diff 和受影响链路
        ↓
DELIVERY_READY
PR Gate 核验证据和交付资格
```

异常状态：

- `NEEDS_USER_DECISION`：业务语义、范围、权限、数据口径或风险接受需要用户决定。
- `BLOCKED`：缺少权限、环境、凭据、fixture 或继续执行所需的外部条件。
- `STALE`：计划、代码或基线在审查期间变化，当前结论失效。
- `STOPPED_INCOMPLETE`：达到内部收敛上限或时限，仍有未关闭问题。

## 计划收敛

### 计划预检输入

- 已确认的 PRD。
- OpenSpec `proposal.md`、`design.md`、`tasks.md` 和相关 specs。
- 验收标准、非目标和风险等级。
- 影响模块和调用链预判。
- 测试策略、迁移策略和回滚方案。

### 固定检查矩阵

- 需求来源与验收标准是否一致。
- 范围、非目标、模块边界是否明确。
- 核心实体、生命周期、状态机和历史事实是否完整。
- API 请求字段是否只表达用户意图，后端事实是否由后端确认。
- DDL、Entity、Request DTO、Response VO、Mapper、Service、测试和文档能否对齐。
- 任务依赖、迁移顺序、验证命令和回滚方式是否可执行。
- 是否遗漏兼容、权限、租户隔离、事务、并发、幂等、性能和外部系统风险。

### 计划结论

- `PLAN_PASS`：计划足以独立执行。
- `PLAN_FIX_REQUIRED`：机械缺口或已确认规格内的不一致，由内部修复后增量复核。
- `NEEDS_USER_DECISION`：存在多种合理业务解释，或修复会改变范围、行为、数据口径和风险接受。

格式、引用、任务遗漏和已确认口径内的跨文档不一致不打断用户。业务语义变化必须由用户确认。

## 首轮综合 Review

首轮 Reviewer 不能发现第一个问题后提前返回。必须完成以下检查矩阵：

```text
需求与验收标准
→ 实际入口和调用链
→ API / DTO / 状态机
→ 事务 / 并发 / 幂等
→ 权限 / 数据隔离
→ Mapper / SQL / 性能
→ 异常路径 / null safety
→ 测试覆盖 / 回滚
→ 范围漂移
```

检查面不适用时必须写明原因；受环境阻塞时必须标记未覆盖，不能假装已检查。

## 统一问题账本

每项问题至少包含：

| 字段 | 说明 |
|---|---|
| `finding_id` | 稳定标识 |
| `severity` | P0 / P1 / P2 / P3 |
| `status` | `ACCEPTED` / `DUPLICATE` / `NOT_AN_ISSUE` / `NEEDS_USER_DECISION` / `OUT_OF_SCOPE` / `RESOLVED` |
| `root_cause` | 根因，不只描述表面现象 |
| `evidence` | 规格、代码、调用链或测试证据 |
| `affected_scope` | 文件、符号、接口、表或调用链 |
| `similarity_scan` | 同类问题应扫描的范围 |
| `fix_boundary` | 允许修改和明确不修改的边界 |
| `required_tests` | 必须新增或重跑的测试 |
| `decision_owner` | 内部可决定或需要用户决定 |

编排层先验证、去重和按根因聚类，再生成修复包。纯风格偏好、没有失败路径的建议和无关历史问题不能阻断当前交付。

## 修复中审查

唯一 Writer 对每个根因执行：

```text
修复根因
+ 扫描同类代码
+ 检查上下游影响
+ 补回归测试
+ 运行 focused tests
+ 自查范围漂移
```

Writer 不能只修改 Reviewer 指出的单一行号。若同类扫描发现当前范围内问题，应并入同一修复包；若属于无关历史问题，应记录为 `OUT_OF_SCOPE`，不扩大任务。

## 增量复审

增量 Reviewer 只检查：

- 原问题是否被真实关闭。
- 修复产生的新 diff。
- 受影响的 callers/callees、API、SQL、事务和状态流转。
- 新增或修改的测试及其执行证据。
- 是否产生回归、范围漂移或新风险。

新问题必须归类：

- `FIX_REGRESSION`：修复引入，立即处理。
- `FIRST_REVIEW_MISS`：首轮已存在且属于当前范围，处理并回流检查矩阵。
- `PRE_EXISTING_OUT_OF_SCOPE`：无关历史问题，记录但不扩大当前任务。
- `STYLE_ONLY`：纯风格偏好，不阻断收敛。

普通任务目标为一个修复包、一次增量复审；最多两次内部 fix loop。A/B 类高风险任务最多三次。超过上限后标记 `STOPPED_INCOMPLETE`，停止盲目修改并请求用户决策。

## 用户体验

### 用户可见节点

用户默认只看到：

1. 开始执行：冻结范围、计划版本、验证边界和未授权动作。
2. 必要决策：无法依据现有规格内部决定的业务、范围、数据或风险问题。
3. 最终结果：实现、测试、内部 Review、已修复问题汇总、剩余风险及交付状态。

内部进行 Review、修复和增量复审时，只提供高层进度，不逐轮展示普通问题清单。

### 用户中断条件

仅在以下情况中断用户：

- 业务规则存在多种合理解释。
- API、权限、数据范围或状态语义需要用户决定。
- 修复将扩大已确认范围或改变对外合同。
- 需要接受不可自动关闭的高风险。
- 缺少继续所需的权限、凭据、环境或外部协调。
- 达到收敛上限或时限仍不能安全完成。

### 最终汇报

```md
## 实现结果

## 验证证据

## 内部审查与修复
- 首轮综合 Review：
- 集中修复：
- 增量复审：

## 剩余风险与未覆盖项

## Git / PR / push / deploy 状态
```

不要求用户阅读每一轮内部问题。用户需要时，可以展开问题账本和审查证据。

## 异常处理

### 工具或环境失败

构建工具、测试环境、CodeGraph、MySQL、第三方 Test Mode 或凭据失败进入独立诊断，不消耗 Review fix loop。恢复后从原固定版本继续；无法恢复时标记 `BLOCKED`。

### 审查期间目标变化

代码、规格或基线变化后，当前结论标记 `STALE`。重新计算固定版本，只对变化范围和受影响链路补审；无法证明变化边界时才重新综合 Review。

### Reviewer 结论冲突

按权威规格、代码路径、测试和运行证据裁决，不按 Reviewer 数量投票。证据不足时标记未确认，不把意见伪装为缺陷。

### dirty worktree

默认保留用户改动。使用 task-only diff/hash 或窄范围 checkpoint；禁止 broad stage。无法安全区分归属时暂停重叠范围。

## 状态与完成条件

只有同时满足以下条件才能进入 `DELIVERY_READY`：

- `PLAN_PASS` 对当前计划 hash 有效。
- 当前范围实现完成。
- 必需测试通过，或未执行项有真实阻塞说明。
- 综合 Review 的成立问题已关闭或获得明确风险接受。
- 增量复审对当前代码 fixed point 有效。
- 没有未处理的范围漂移。
- PR Gate 所需证据齐全。
- push、PR、部署和数据库操作状态被准确区分。

## 度量指标

不以 Review 次数或发现问题数量作为成绩。观察：

- 首轮问题覆盖率。
- 平均内部 fix loop 数。
- `FIRST_REVIEW_MISS` 数量和类型。
- 同类问题重复出现率。
- 用户被中断次数。
- Review 误报率。
- 修复引入的回归数。
- PR Gate 首次发现普通问题的数量。

初始目标：

- 普通任务平均内部 fix loop 不超过一次。
- 普通技术问题不打断用户。
- PR Gate 不再作为普通缺陷的主要发现点。
- 同类问题第二次出现时，评估升级检查矩阵、checklist、AGENTS 或 Skill。

## 验证方案

实现后至少用以下场景验证编排规则：

1. C 类单文件 bugfix：应走一次综合 Review 和一次增量复审，不触发专项 Reviewer。
2. 普通跨层需求：Java、SQL、测试问题进入一个问题账本，并按根因形成一个修复包。
3. A 类高风险需求：按风险触发专项检查，但仍由编排层统一向用户汇报。
4. 业务语义不清：进入 `NEEDS_USER_DECISION`，不得自行猜测。
5. Reviewer 误报：经证据裁决后标记 `NOT_AN_ISSUE`，不得交给 Writer 机械修改。
6. 修复引入回归：增量复审识别 `FIX_REGRESSION`，修复后重新验证。
7. 无关历史问题：标记 `PRE_EXISTING_OUT_OF_SCOPE`，不扩大当前任务。
8. 审查期间代码变化：旧结论标记 `STALE`，只补审可证明的变化范围。
9. dirty worktree：证明 task-only 范围，没有混入用户改动。
10. 超过收敛上限：标记 `STOPPED_INCOMPLETE`，向用户提供剩余问题和决策点。

## 分阶段落地

### 第一阶段：规则接入

- 新增 `delivery-convergence-loop` Skill。
- 调整 `asana-openspec-delivery`，在计划冻结后进入收敛编排。
- 调整各专业 Reviewer 的输出合同，使结果进入统一问题账本。
- 调整 `pr-quality-gate`，强调最终证据验收职责。

### 第二阶段：真实任务试运行

- 选择一个 C 类 bugfix。
- 选择一个普通跨层需求。
- 选择一个 A/B 类高风险 slice。
- 记录首轮覆盖率、fix loop、用户中断、误报和 PR Gate 新发现问题。

### 第三阶段：规则回流

- 重复漏检升级检查矩阵。
- 重复修复模式升级 Writer 同类扫描规则。
- 无价值字段删除或改为条件字段。
- 指标稳定后再考虑自动化审计或更强硬门禁。

## 验收标准

- 普通需求不再把每轮 Review 问题逐项暴露给用户。
- 首轮综合 Review 有固定检查矩阵和完整性声明。
- 多个专业 Skill 的结果能被去重并形成统一问题账本。
- Writer 按根因修复，并提供同类扫描和回归测试证据。
- 修复后默认执行增量复审，而不是重新全量盲审。
- 普通任务最多两次内部 fix loop，高风险任务最多三次。
- PR Gate 主要核验交付证据，不重复承担普通缺陷发现。
- 业务、范围和高风险决策仍由用户确认，没有因静默闭环而越权。
