## 文档信息

| 项目 | 内容 |

| 日期 | 2026-07-20 |

## 背景

当前工作流已有 PRD Review、Plan Review、Java 后端评审、SQL 性能评审、安全评审、测试策略和 PR Quality Gate。各门禁能发现问题，但如果每个检查面独立返回、每轮修复后重新全量审查，用户会持续看到“发现问题—修复—又发现问题”的过程。

问题不在于发现问题太多，而在于：

- 首轮审查缺少统一输入合同和完整检查矩阵，容易挤牙膏式发现问题。
- 不同专业 Skill 分别向用户汇报，缺少问题去重、归因和集中修复。
- Writer 只修单点，没有强制扫描同类位置、上下游影响和回归测试。
- 修复后重新全量盲审，重复成本高，也容易扩大审查范围。
- PR Gate 承担了过多普通缺陷发现职责，而不是最终证据验收职责。
- 内部质量过程直接暴露给用户，用户被迫参与技术修复调度。

## 目标

新增交付收敛编排层，在不削弱现有专业门禁的前提下，实现：

1. 计划阶段一次完整预检，先消除结构性缺口再开发。
2. 代码阶段一次综合深审，形成完整、去重的问题账本。
3. 按根因集中修复，并在修复过程中扫描同类问题和上下游影响。
4. 后续只做增量复审，不机械重复全量审查。
5. 普通问题在内部收敛，用户只处理业务决策、范围变化和剩余高风险。
6. 普通任务目标为一个修复包完成收敛，平均内部 fix loop 不超过一次。

## 非目标

- 不降低 P0-P3 成立问题的修复标准。
- 不用“限制 Reviewer 报问题数量”制造虚假通过。
- 不让编排层替代 PRD、OpenSpec、CodeGraph、测试或专业 Review Skill。
- 不让 Reviewer 接手生产代码修改。
- 不强制所有任务使用多个 Reviewer、worktree 或自动 checkpoint commit。
- 不把无关历史问题带入当前需求，造成范围无限扩张。
- 不自动执行 push、PR、部署、生产操作或数据库变更。

## 核心原则

### 1. 审查是内部交付能力

内部可以存在 Review、修复和复审，但用户默认只看到范围确认、必要决策和最终结果。能依据已确认规格自行修复的问题，不要求用户逐轮确认。

### 2. 首轮看全，后续看差异

首轮 Reviewer 必须完成固定检查矩阵，并明确未覆盖项。修复后的 Reviewer 只检查问题关闭情况、新增 diff、受影响调用链和回归证据。

### 3. 按根因修复，不按评论逐条改

多个表现属于同一根因时合并成一个修复包。Writer 修复根因时同步扫描同类代码、上下游影响和测试缺口。

### 4. 风险决定审查强度

- `light` 路径使用轻量综合 Review：只启用适用检查面；没有成立问题时直接进入 PR Gate，有修复才执行一次增量复审。
- `standard` 路径使用一次完整综合 Review；没有成立问题时直接进入 PR Gate，有修复才执行一次增量复审。
- `high_risk` 路径按命中条件启用专项检查；专项结果仍合并为一次综合 Review 输出，不分别打断用户。

### 5. 结论绑定固定版本

计划结论绑定 PRD/OpenSpec/tasks 内容 hash；代码结论绑定 Git SHA。无法提交时绑定 task-only diff hash。目标版本变化后旧结论只作为历史证据，不直接作为新版本通过依据。

## 总体架构

新增编排 Skill，正式名称为 `delivery-convergence-loop`，目录和 frontmatter `name` 均使用该名称。它不直接判断 Java、SQL、安全或测试细节，只负责固定输入、调度专业 Skill、维护问题账本、控制修复和复审、管理用户可见性。

```text
已确认 PRD / OpenSpec / tasks
        ↓
PLAN_PREFLIGHT
计划一次性预检
        ↓
DEV_ACTIVE
开发、测试、自检
        ↓
COMPREHENSIVE_REVIEW
合并规格、Java、SQL、安全、测试检查面
        ├─ PASS ──────────────────────→ PR_GATE
        └─ FIX_REQUIRED → FIXING
                          集中修复、同类扫描、回归测试
                                ↓
                          INCREMENTAL_REVIEW
                          只复审修复点、新 diff 和受影响链路
                                ↓
                          REVIEW_PASS → PR_GATE
PR_GATE → DELIVERY_READY / DELIVERY_CONDITIONAL / BLOCKED
```

异常状态：

- `NEEDS_USER_DECISION`：业务语义、范围、权限、数据口径或风险接受需要用户决定。
- `BLOCKED`：缺少权限、环境、凭据、fixture 或继续执行所需的外部条件。
- `STALE`：计划、代码或基线在审查期间变化，当前结论失效。
- `STOPPED_INCOMPLETE`：达到内部收敛上限或时限，仍有未关闭问题。

### 状态转移

```text
INTAKE_READY
→ PLAN_PREFLIGHT
→ PLAN_PASS
→ DEV_ACTIVE
→ SELF_CHECKED
→ COMPREHENSIVE_REVIEW
→ REVIEW_PASS ───────────────→ PR_GATE
             └─有成立问题→ FIXING
                              → INCREMENTAL_REVIEW
                              → REVIEW_PASS 或下一次 FIXING
PR_GATE → DELIVERY_READY / DELIVERY_CONDITIONAL / BLOCKED
```

恢复规则：

- `NEEDS_USER_DECISION`：保存进入前状态；用户决定后更新权威规格或风险记录，重新计算 fixed point，并从最早受影响阶段恢复。
- `BLOCKED`：外部条件恢复后重新核对 fixed point；未变化则返回原阶段，已变化则先进入 `STALE`。
- `STALE`：生成新 fixed point 和影响差异；能证明变化边界时从受影响检查面恢复，否则回到综合 Review。
- `STOPPED_INCOMPLETE`：本次运行停止，不自动授予通过。用户继续时创建续跑记录，继承历史证据但重新验证当前 fixed point。
- `DELIVERY_CONDITIONAL`：仅表示 PR Gate 按现有规则给出 `CONDITIONAL`，不能表达已完全通过。
- `DELIVERY_READY`：仅当 PR Gate 为 `PASS` 且所有必需验证完成时成立。

### 风险等级与交付路径

风险等级和交付路径是两个独立字段：

- `risk_class`：沿用现有协议，只允许 `A` / `B` / `C`。
- `delivery_path`：只允许 `light` / `standard` / `high_risk`。

确定性映射：

- C 类默认 `light`；负责人或检查面触发升级时可进入 `standard`，不能自动降级。
- B 类默认 `standard`；命中专项高风险检查条件或负责人升级时进入 `high_risk`。
- A 类固定 `high_risk`。

`delivery_path` 在 `PLAN_PASS` 前冻结并记录判定依据。后续只能因发现更高风险而升级；降级必须获得项目负责人明确确认并重新执行 Plan Preflight。

### 唯一调度与适配器

进入收敛阶段后，`delivery-convergence-loop` 是专业 Review Skill 的唯一调度者。被调度 Skill 不再嵌套调用已经由本轮编排覆盖的其他 Review Skill；确需建议另一个检查面时，只返回触发建议，由编排层决定是否调用。这样避免 `java-backend-review`、主交付 Skill 和专项 Skill 重复触发同一检查。

专业 Skill 保留原始输出合同。编排层通过 adapter 转成统一 `ReviewResult`，不得要求专业 Skill 丢弃其原有四轴、严重度或结论结构。

adapter 必须：

- 保存原始结果引用和来源 Skill。
- 映射严重度、证据、检查面、结论和未覆盖项。
- 使用 `(requirement_ref, affected_scope, root_cause, failure_mode)` 作为候选去重键，再由证据裁决是否真正重复。
- 合并重复项时保留全部 `source_skill` 和原始证据引用。
- adapter 输出缺少必填字段时标记 `ADAPTER_INVALID`，不能进入修复或通过状态。

## Fixed point 规范

### 计划 fixed point

计划 fixed point 使用 SHA-256。输入为本轮声明的权威计划文件清单，至少包含适用的 PRD、OpenSpec `proposal.md`、`design.md`、`tasks.md` 和 delta specs。

规范化算法：

1. 路径转为仓库相对路径并使用 `/`，按 ordinal 升序排列。
2. UTF-8 文本移除 BOM，将 CRLF/CR 统一为 LF；不裁剪空格和结尾换行。
3. 每个文件记录固定字段 `{"content_sha256": string, "length": integer, "path": string}`；文件数组按 `path` ordinal 升序排列。
4. manifest 固定为 `{"files": [...], "schema_version": 1}`，按 RFC 8785 JSON Canonicalization Scheme 序列化为 UTF-8，再计算 SHA-256，得到 `plan_fixed_point`。

权威文件增删、内容变化或清单变化都会使原 `PLAN_PASS` 过期。

### 代码 fixed point

代码 fixed point 必须先冻结 task scope 白名单。它由以下字段组成：

- `HEAD_SHA` 和相关 submodule 状态。submodule 记录固定字段 `{"head": string, "path": string, "status": "clean|modified|uninitialized|conflict"}`，按 `path` ordinal 排序。
- task scope 内 tracked 文件的 staged binary patch：`git diff --cached --binary --no-ext-diff`。
- task scope 内 tracked 文件的 unstaged binary patch：`git diff --binary --no-ext-diff`。
- task scope 内 untracked 文件的仓库相对路径、原始字节长度和 SHA-256。
- task scope 白名单本身。

task scope 路径按 ordinal 排序，并作为独立 argv 元素放在 `--` 后传给 Git，禁止通过 shell 字符串拼接或 glob 展开。patch 使用以下固定选项生成：`--binary --no-ext-diff --no-textconv --no-renames --full-index --src-prefix=a/ --dst-prefix=b/`；staged patch 额外使用 `--cached`。patch 保留 Git 原始字节，不做换行转换。

代码 manifest 固定为：

```json
{
  "schema_version": 1,
  "head_sha": "...",
  "task_scope": ["..."],
  "submodules": [],
  "staged_patch_sha256": "...",
  "unstaged_patch_sha256": "...",
  "untracked_files": []
}
```

`untracked_files` 元素固定为 `{"content_sha256": string, "length": integer, "path": string}`。数组均按路径 ordinal 排序，字符串使用 JSON 标准转义；manifest 按 RFC 8785 序列化。最终 hash 输入为：JCS manifest UTF-8 字节、staged patch 长度（8 字节 big-endian）及字节、unstaged patch 长度及字节，按该顺序拼接后计算 SHA-256，得到 `code_fixed_point`。task scope 外的 dirty 文件不进入 hash，但必须记录在运行状态中，证明它们没有被纳入修复、暂存或提交。

无法安全冻结 task scope，或同一 hunk 混有无法归属的用户改动时，进入 `BLOCKED`，不得用全仓库 broad stage 代替。

## 计划收敛

### 计划预检输入

- 已确认的 PRD。
- OpenSpec `proposal.md`、`design.md`、`tasks.md` 和相关 specs。
- 验收标准、非目标和风险等级。
- 影响模块和调用链预判。
- 测试策略、迁移策略和回滚方案。

### 固定检查矩阵

- 需求来源与验收标准是否一致。
- 范围、非目标、模块边界是否明确。
- 核心实体、生命周期、状态机和历史事实是否完整。
- API 请求字段是否只表达用户意图，后端事实是否由后端确认。
- DDL、Entity、Request DTO、Response VO、Mapper、Service、测试和文档能否对齐。
- 任务依赖、迁移顺序、验证命令和回滚方式是否可执行。
- 是否遗漏兼容、权限、租户隔离、事务、并发、幂等、性能和外部系统风险。

### 计划结论

- `PLAN_PASS`：计划足以独立执行。
- `PLAN_FIX_REQUIRED`：机械缺口或已确认规格内的不一致，由内部修复后增量复核。
- `NEEDS_USER_DECISION`：存在多种合理业务解释，或修复会改变范围、行为、数据口径和风险接受。

格式、引用、任务遗漏和已确认口径内的跨文档不一致不打断用户。业务语义变化必须由用户确认。

## 首轮综合 Review

首轮 Reviewer 不能发现第一个问题后提前返回。必须完成以下检查矩阵：

```text
需求与验收标准
→ 实际入口和调用链
→ API / DTO / 状态机
→ 事务 / 并发 / 幂等
→ 权限 / 数据隔离
→ Mapper / SQL / 性能
→ 异常路径 / null safety
→ 测试覆盖 / 回滚
→ 范围漂移
```

检查面不适用时必须写明原因；受环境阻塞时必须标记未覆盖，不能假装已检查。

### `ReviewInput`

| 字段 | 说明 |
|---|---|
| `run_id` | 本次收敛运行标识 |
| `plan_fixed_point` | 已通过的计划 hash |
| `code_fixed_point` | 待审代码 fixed point |
| `scope` / `non_goals` | 允许范围和明确非目标 |
| `risk_class` | A / B / C |
| `delivery_path` | `light` / `standard` / `high_risk` |
| `acceptance_criteria` | 可逐项验证的验收标准 |
| `change_map` | 入口、调用链、文件、接口和表的改动地图 |
| `test_evidence` | 已执行测试及未执行项 |
| `required_review_surfaces` | 本轮应启用的检查面 |

### `ReviewResult`

| 字段 | 说明 |
|---|---|
| `reviewed_code_fixed_point` | 实际审查版本 |
| `coverage_declaration` | 每个检查面为 `COVERED` / `NOT_APPLICABLE` / `BLOCKED`，附理由 |
| `raw_result_refs` | 专业 Skill 原始输出引用 |
| `findings` | adapter 转换后的问题列表 |
| `uncovered_items` | 未完成的检查和原因 |
| `result` | `PASS` / `FIX_REQUIRED` / `NEEDS_USER_DECISION` / `BLOCKED` |

`coverage_declaration` 未覆盖全部必需检查面时不得返回 `PASS`。

## 统一问题账本

每项问题至少包含：

| 字段 | 说明 |
|---|---|
| `finding_id` | 稳定标识 |
| `severity` | P0 / P1 / P2 / P3 |
| `finding_type` | `DEFECT` / `FIX_REGRESSION` / `FIRST_REVIEW_MISS` / `PRE_EXISTING_OUT_OF_SCOPE` / `STYLE_ONLY` |
| `status` | `OPEN` / `VALIDATED` / `FIXING` / `RESOLVED` / `VERIFIED` / `WAIVED` / `DUPLICATE` / `REJECTED` / `OUT_OF_SCOPE` / `NEEDS_DECISION` |
| `root_cause` | 根因，不只描述表面现象 |
| `evidence` | 规格、代码、调用链或测试证据 |
| `affected_scope` | 文件、符号、接口、表或调用链 |
| `similarity_scan` | 同类问题应扫描的范围 |
| `fix_boundary` | 允许修改和明确不修改的边界 |
| `required_tests` | 必须新增或重跑的测试 |
| `decision_owner_role` | `NONE` / `USER` / `BUSINESS_OWNER` / `TECH_OWNER` / `SECURITY_OWNER` / `DB_OWNER` |
| `decision_authority_ref` | 角色授权依据，对应 AGENTS、PRD、OpenSpec 或项目治理规则 |
| `source_skills` | 来源 Skill 及原始结果引用 |
| `introduced_at` | 首次出现的 code fixed point |
| `resolved_at` | 修复后的 code fixed point |
| `waiver` | 接受人、时间、依据、补偿措施和有效范围 |

生命周期：

- 主修复链：`OPEN → VALIDATED → FIXING → RESOLVED → VERIFIED`。
- 初审分支：`OPEN → REJECTED / DUPLICATE / OUT_OF_SCOPE / NEEDS_DECISION`。
- 决策后修复：`NEEDS_DECISION → VALIDATED → FIXING`。
- 授权接受：`NEEDS_DECISION → WAIVED`，或 `VALIDATED → WAIVED`。
- 增量复审失败：`RESOLVED → FIXING`，并在新修复批次中处理。

不成立进入 `REJECTED`；重复项进入 `DUPLICATE` 并指向主问题；无关历史问题进入 `OUT_OF_SCOPE`。`WAIVED` 只用于获得授权的风险接受，不能由编排层自行授予。

编排层先验证、去重和按根因聚类，再生成修复包。纯风格偏好、没有失败路径的建议和无关历史问题不能阻断当前交付。

### 风险接受权限

- P0/P1 必须修复，编排层、Writer 和 Reviewer 均无权静默接受。
- P2 必须修复，或按现有 PR Gate 规则取得 `USER`、`BUSINESS_OWNER`、`TECH_OWNER`、`SECURITY_OWNER` 或 `DB_OWNER` 中具备项目授权的角色明确决定；接受后整体最多进入 `DELIVERY_CONDITIONAL`。
- 当前范围内成立的 P3 默认修复。只有证明不属于验收范围或修改成本明显超出当前边界时，才能转为 `OUT_OF_SCOPE`，或由具备项目授权的 `TECH_OWNER`/对应领域 Owner/`USER` 明确接受。
- 所有 `WAIVED` 必须记录接受人、时间、依据、补偿措施、未覆盖风险和适用 fixed point。项目 `AGENTS.md`、PRD、OpenSpec 和 PR Gate 有更严格规则时，以更严格规则为准。
- `decision_authority_ref` 为空或无法证明角色权限时，保持 `NEEDS_DECISION` 并询问用户，不能执行 `WAIVED` 转移。

## 修复中审查

一次综合或增量 `ReviewResult` 产生一个 `FixBatch`。一个批次可以包含多个按根因拆分的 `FixPackage`；同一批次的所有 package 共用一个 `loop_index`，避免多个根因被误计为多轮。

`FixBatch` 必填字段：`batch_id`、`source_review_result_id`、`base_code_fixed_point`、`package_ids`、`loop_index`、`deadline_at` 和整体结果。

### `FixPackage`

| 字段 | 说明 |
|---|---|
| `package_id` | 修复包标识 |
| `batch_id` | 所属修复批次 |
| `base_code_fixed_point` | 修复开始前版本 |
| `finding_ids` | 按同一根因聚类的问题 |
| `root_cause` | 本包要消除的根因 |
| `allowed_scope` / `forbidden_scope` | 可改范围和禁改范围 |
| `similarity_scan` | 必须扫描的同类范围 |
| `required_tests` | 必须新增或执行的测试 |
| `loop_index` | 继承所属批次的 fix loop 序号 |

唯一 Writer 对每个根因执行：

```text
修复根因
+ 扫描同类代码
+ 检查上下游影响
+ 补回归测试
+ 运行 focused tests
+ 自查范围漂移
```

Writer 不能只修改 Reviewer 指出的单一行号。若同类扫描发现当前范围内问题，应并入同一修复包；若属于无关历史问题，应记录为 `OUT_OF_SCOPE`，不扩大任务。

### `FixResult`

| 字段 | 说明 |
|---|---|
| `package_id` / `batch_id` | 对应修复包和批次 |
| `new_code_fixed_point` | 修复后的版本 |
| `finding_resolution_map` | 每个 finding 的修改位置和关闭依据 |
| `changed_scope` | 实际变更文件、符号、接口和表 |
| `similarity_scan_result` | 扫描范围、命中和处置 |
| `test_evidence` | focused tests、失败信息和未执行项 |
| `scope_deviation` | 是否偏离修复边界 |
| `result` | `FIXED` / `FIX_FAILED` / `TEST_FAILED` / `SCOPE_DEVIATION` / `BLOCKED` |

## 增量复审

增量复审输入必须包含：上一版 `ReviewResult`、固定的账本快照 ID、`FixPackage`、`FixResult`、旧/新 code fixed point 间的 task-only diff，以及受影响调用链和测试证据。

增量 Reviewer 只检查：

- 原问题是否被真实关闭。
- 修复产生的新 diff。
- 受影响的 callers/callees、API、SQL、事务和状态流转。
- 新增或修改的测试及其执行证据。
- 是否产生回归、范围漂移或新风险。

新问题必须归类：

- `FIX_REGRESSION`：修复引入，立即处理。
- `FIRST_REVIEW_MISS`：首轮已存在且属于当前范围，处理并回流检查矩阵。
- `PRE_EXISTING_OUT_OF_SCOPE`：无关历史问题，记录但不扩大当前任务。
- `STYLE_ONLY`：纯风格偏好，不阻断收敛。

### `IncrementalReviewResult`

| 字段 | 说明 |
|---|---|
| `result_id` | 本次增量复审标识 |
| `reviewed_code_fixed_point` | 实际复审的新版本 |
| `base_ledger_snapshot_id` | 输入账本快照 |
| `new_ledger_snapshot_id` | 输出账本快照，不覆盖历史快照 |
| `verified_findings` | 转为 `VERIFIED` 的 finding IDs |
| `new_findings` | 新增问题、`finding_type` 和证据 |
| `unresolved_findings` | 仍未关闭的问题 |
| `test_evidence` | 本轮使用的测试证据 |
| `scope_deviation` | 范围漂移结论和证据 |
| `result` | `PASS` / `FIX_REQUIRED` / `NEEDS_USER_DECISION` / `BLOCKED` / `STALE` / `STOPPED_INCOMPLETE` |
| `next_state` | 确定的下一状态 |

结果映射：

- `PASS → REVIEW_PASS`。
- `FIX_REQUIRED → FIXING`，并生成新的 `FixBatch`。
- `NEEDS_USER_DECISION → NEEDS_USER_DECISION`。
- `BLOCKED → BLOCKED`。
- `STALE → STALE`，按 fixed point 恢复规则处理。
- `STOPPED_INCOMPLETE → STOPPED_INCOMPLETE`。

### 轮次与时限

- 初始功能实现不计入 fix loop。
- Reviewer 生成 `FixBatch` 且 Writer 首次开始修改生产代码时，`loop_index` 增加一次；同一批次中的多个根因、多个 package、诊断和 focused test 重跑不重复计数。
- `FIRST_REVIEW_MISS` 或 `FIX_REGRESSION` 经增量复审成立，并生成新 `FixBatch` 时，计入下一次 fix loop。
- 外部修改导致 `STALE`、等待用户决定、环境恢复和 adapter 重试不计入 fix loop，但可能使旧证据失效。
- `light` 和 `standard` 路径最多两次 fix loop；`high_risk` 路径最多三次。
- 每次运行在 `INTAKE_READY` 时固定 `deadline_at`。用户未指定时，`light` 和 `standard` 默认 60 分钟，`high_risk` 默认 90 分钟；等待和重试不重置截止时间。
- 单次专业 Review Skill 调用的超时为“20 分钟”和当前 `deadline_at` 剩余时间中的较小值；用户明确指定更短单次上限时，以用户值为准。
- 达到轮次或时间上限后标记 `STOPPED_INCOMPLETE`，停止盲目修改并请求用户决策。

## 用户体验

### 用户可见节点

用户默认只看到：

1. 开始执行：冻结范围、计划版本、验证边界和未授权动作。
2. 必要决策：无法依据现有规格内部决定的业务、范围、数据或风险问题。
3. 最终结果：实现、测试、内部 Review、已修复问题汇总、剩余风险及交付状态。

内部进行 Review、修复和增量复审时，只提供高层进度，不逐轮展示普通问题清单。

### 用户中断条件

仅在以下情况中断用户：

- 业务规则存在多种合理解释。
- API、权限、数据范围或状态语义需要用户决定。
- 修复将扩大已确认范围或改变对外合同。
- 需要接受不可自动关闭的高风险。
- 缺少继续所需的权限、凭据、环境或外部协调。
- 达到收敛上限或时限仍不能安全完成。

### 最终汇报

```md
## 实现结果

## 验证证据

## 内部审查与修复
- 首轮综合 Review：
- 集中修复：
- 增量复审：

## 剩余风险与未覆盖项

## Git / PR / push / deploy 状态
```

不要求用户阅读每一轮内部问题。用户需要时，可以展开问题账本和审查证据。

## 异常处理

### 工具或环境失败

构建工具、测试环境、CodeGraph、MySQL、第三方 Test Mode 或凭据失败进入独立诊断，不消耗 Review fix loop。恢复后从原固定版本继续；无法恢复时标记 `BLOCKED`。

### 专业 Review 或 adapter 失败

- 专业 Skill 达到单次调用超时或只返回部分检查面：保留成功结果，将缺失面标记 `BLOCKED`，只重试缺失面；不得把部分成功合并成 `PASS`。达到全局 `deadline_at` 时转为 `STOPPED_INCOMPLETE`。
- adapter 缺少必填字段：标记 `ADAPTER_INVALID`，允许基于原始输出修正 adapter 并重试一次；仍失败则进入 `BLOCKED`。
- 专业 Skill 返回互相冲突的结论：进入证据裁决，未决项标记 `NEEDS_DECISION` 或 `BLOCKED`，不得按票数合并。

### Writer 或测试失败

- `FIX_FAILED`：保持问题为 `FIXING`，先诊断失败原因；同一修复包内继续不增加 loop。
- `TEST_FAILED`：区分原有失败、环境失败和本次回归。本次回归必须留在 `FIXING`；环境失败按工具失败处理。
- `SCOPE_DEVIATION`：停止 Writer，恢复到安全 fixed point 或拆分新范围；涉及业务范围变化时进入 `NEEDS_USER_DECISION`。
- Writer 无法在当前修复包内安全完成，或重复失败达到 `deadline_at`：进入 `STOPPED_INCOMPLETE`。

### 审查期间目标变化

代码、规格或基线变化后，当前结论标记 `STALE`。重新计算固定版本，只对变化范围和受影响链路补审；无法证明变化边界时才重新综合 Review。

### Reviewer 结论冲突

按权威规格、代码路径、测试和运行证据裁决，不按 Reviewer 数量投票。证据不足时标记未确认，不把意见伪装为缺陷。

### dirty worktree

默认保留用户改动。使用 task-only diff/hash 或窄范围 checkpoint；禁止 broad stage。无法安全区分归属时暂停重叠范围。

## 运行状态持久化

每次运行开始时必须解析 `RUN_STATE_DIR`：优先使用宿主提供的仓库外持久任务目录；宿主未提供时，只有项目明确允许且路径已被 Git 忽略，才使用仓库内 `.codex/delivery-runs/<run-id>/`。两者都不可用且任务可能跨上下文时，进入 `BLOCKED`，不得声称可恢复续跑。

最小结构：

```text
<RUN_STATE_DIR>/
  运行状态.json
  fixed-points/
    计划版本.json
    代码版本.json
  审查原始结果/
  问题账本.jsonl
  修复包/
  测试证据索引.json
```

要求：

- `运行状态.json` 保存当前状态、进入前状态、`risk_class`、`delivery_path`、路径判定依据、`loop_index`、`deadline_at`、task scope 和最近账本快照。
- 账本和 fixed point 采用追加或版本化写入，禁止覆盖历史审查证据。
- 状态更新使用临时文件加原子重命名；写入失败时保持上一有效快照并进入 `BLOCKED`。
- 凭据、Token、完整敏感请求体和生产数据不得写入运行状态；证据只保存脱敏摘要和安全引用。
- 运行状态保留到用户接受交付结果或项目归档节点；是否清理由宿主或项目策略决定，编排层不自动删除。

## 状态与完成条件

只有同时满足以下条件才能进入 `DELIVERY_READY`：

- `PLAN_PASS` 对当前计划 hash 有效。
- 当前范围实现完成。
- 所有必需测试通过；存在未执行的必需测试时只能进入 `BLOCKED` 或按现有 PR Gate 规则进入 `DELIVERY_CONDITIONAL`，不能进入 `DELIVERY_READY`。
- 综合 Review 的 P0/P1 已修复，P2/P3 已按风险接受权限处理。
- 当前代码 fixed point 有有效 Review 通过证据：若综合 Review 后未修改代码，使用该 fixed point 的综合 `PASS`；若综合 Review 后发生修复，必须使用新 fixed point 的增量复审 `PASS`。
- 没有未处理的范围漂移。
- PR Gate 返回 `PASS`；返回 `CONDITIONAL` 时只能进入 `DELIVERY_CONDITIONAL`。
- push、PR、部署和数据库操作状态被准确区分。

## 度量指标

不以 Review 次数或发现问题数量作为成绩。观察：

- 首轮问题覆盖率：`首轮 VALIDATED findings /（首轮 VALIDATED + 后续 FIRST_REVIEW_MISS）`。仅在本次运行最终问题集稳定后计算。
- 平均内部 fix loop：已完成运行的 `loop_index` 总和 / 已完成运行数。
- 同类问题重复出现率：被标记为已回流规则、但后续运行再次出现的同类根因数 / 已回流根因数。
- 用户被中断次数：进入 `NEEDS_USER_DECISION` 并实际发送用户决策请求的次数。
- Review 误报率：`REJECTED findings / 所有完成证据裁决的 findings`。
- 修复引入回归数：`FIX_REGRESSION` 数量。
- PR Gate 首次发现普通问题数：未出现在综合 Review 账本、由 PR Gate 首次发现的 P2/P3 数量。

首轮覆盖率和平均 fix loop 在试运行阶段只作为观察指标，不作为单次交付 PASS 的硬门槛。

初始目标：

- 普通任务平均内部 fix loop 不超过一次。
- 普通技术问题不打断用户。
- PR Gate 不再作为普通缺陷的主要发现点。
- 同类问题第二次出现时，评估升级检查矩阵、checklist、AGENTS 或 Skill。

## 验证方案

实现后至少用以下场景验证编排规则：

1. C 类单文件 bugfix：应走轻量综合 Review，不触发专项 Reviewer；只有 Review 后发生修复时才执行一次增量复审。
2. 普通跨层需求：Java、SQL、测试问题进入一个问题账本，并按根因形成一个修复包。
3. A 类高风险需求：按风险触发专项检查，但仍由编排层统一向用户汇报。
4. 业务语义不清：进入 `NEEDS_USER_DECISION`，不得自行猜测。
5. Reviewer 误报：经证据裁决后标记 `REJECTED`，不得交给 Writer 机械修改。
6. 修复引入回归：增量复审识别 `FIX_REGRESSION`，修复后重新验证。
7. 无关历史问题：标记 `PRE_EXISTING_OUT_OF_SCOPE`，不扩大当前任务。
8. 审查期间代码变化：旧结论标记 `STALE`，只补审可证明的变化范围。
9. dirty worktree：证明 task-only 范围，没有混入用户改动。
10. 超过收敛上限：标记 `STOPPED_INCOMPLETE`，向用户提供剩余问题和决策点。

## 分阶段落地

### 第一阶段：规则接入

- 新增 `delivery-convergence-loop` Skill。
- 调整 `asana-openspec-delivery`，在计划冻结后进入收敛编排。
- 为各专业 Reviewer 增加 adapter，保留原始输出合同并转换为统一问题账本。
- 增加 fixed point、运行状态、问题账本、修复包和增量复审合同。
- 明确收敛阶段由编排层唯一调度专业 Review，禁止嵌套重复调用。
- 调整 `pr-quality-gate`，强调最终证据验收职责。

### 第二阶段：真实任务试运行

- 选择一个 C 类 bugfix。
- 选择一个普通跨层需求。
- 选择一个 A 类高风险或 B 类复杂 slice。
- 记录首轮覆盖率、fix loop、用户中断、误报和 PR Gate 新发现问题。

### 第三阶段：规则回流

- 重复漏检升级检查矩阵。
- 重复修复模式升级 Writer 同类扫描规则。
- 无价值字段删除或改为条件字段。
- 指标稳定后再考虑自动化审计或更强硬门禁。

## 验收标准

- 普通需求不再把每轮 Review 问题逐项暴露给用户。
- 首轮综合 Review 有固定检查矩阵和完整性声明。
- 多个专业 Skill 的结果能被去重并形成统一问题账本。
- 专业 Skill 原始输出被保留，adapter 转换失败不能产生通过结论。
- 计划和代码 fixed point 可按规范化算法复现，目标变化能可靠触发 `STALE`。
- Writer 按根因修复，并提供同类扫描和回归测试证据。
- 修复后默认执行增量复审，而不是重新全量盲审。
- 普通任务最多两次内部 fix loop，高风险任务最多三次。
- 运行状态、账本快照、修复包和证据可以跨上下文恢复；无法持久化时不得声称可续跑。
- PR Gate 主要核验交付证据，不重复承担普通缺陷发现。
- 业务、范围和高风险决策仍由用户确认，没有因静默闭环而越权。
