# Asana OpenSpec Java Workflow

这是一个轻量 Codex 插件包，用来把团队需求从 Asana 推进到 PRD、OpenSpec change、Java 实现、MySQL 数据验证、质量门禁和 PR。

## 适用团队

- 需求入口在 Asana。
- 代码主要是 Java、Spring Boot、MyBatis、SQL。
- 希望用 OpenSpec 记录需求变更，而不是只靠聊天上下文。
- 希望 PR 前有固定的 Java 后端 Review、MySQL 数据安全规则和质量门禁。

## 插件内容

- `prd-writer`：把 Asana 需求整理成 PRD。
- `asana-openspec-delivery`：主交付流程。
- `large-refactor-workflow`：大重构升级流程，按 Epic、RFC、phase/tasks、多 agent 评估、小 PR 串行推进。
- `codegraph-context-guard`：CodeGraph 知识图谱使用规则，防止 AI 乱定位代码。
- `coding-discipline`：AI 写代码纪律，吸收 Karpathy 的少猜、小改、可验证原则。
- `java-coding-standard`：Java 编码规范，约束命名、分层、异常、日志、MyBatis、测试。
- `springboot-service-patterns`：Spring Boot 服务设计模式，约束 Controller/Service/Mapper、事务、分页、缓存、异步。
- `springboot-security-review`：Spring Boot 安全评审，检查认证、授权、输入校验、SQL 注入、密钥、敏感日志。
- `java-build-fix`：Java/Maven/Gradle 构建失败修复流程。
- `java-test-strategy`：Java 测试策略，规定单测、集成测试、手动验证和 PR 测试说明。
- `java-backend-review`：Java/Spring/MyBatis 后端专项评审。
- `mysql-db-guard`：MySQL MCP / AI 专用账号的读写安全规则。
- `pr-quality-gate`：PR 前质量门禁。
- 可选 Superpowers：`superpowers:brainstorming` 用于需求澄清，`superpowers:writing-plans` 用于复杂实现拆计划；不替代 PRD / OpenSpec 主流程。
- `assets/templates/`：PRD、Asana 字段、OpenSpec、MySQL、PR Review、AGENTS 模板、多 Agent 任务单、Claude 审查任务单和结构化状态模板。

## 推荐落地方式

1. 先按 `团队安装指南.md` 把本工具包 clone 到 Codex plugins 目录。
2. 再按 `安装配置指南.md` 安装和启用 OpenSpec、CodeGraph、Asana/GitHub Connector、MySQL MCP 等外部工具。
3. 先看 `流程图.md`，确认团队理解“小需求”和“大重构”的分流方式。
4. 在 Asana 建需求项目和字段，参考 `assets/templates/Asana字段模板.md`。
5. 在 Java 项目仓库初始化 OpenSpec。
6. 把 `assets/templates/AGENTS模板.md` 内容合并到项目 `AGENTS.md`。
7. 为 MySQL MCP 准备一个专用 AI 账号，参考 `assets/templates/MySQL数据库安全模板.md`。
8. 每条需求先判断是小需求还是大重构。
9. 如果需求模糊、新功能探索或方案分歧较大，可先用 `superpowers:brainstorming` 辅助澄清；输出只作为 PRD 输入。
10. 小需求：先产出 PRD，再检查既有 `openspec/specs/*`、active changes、历史 PRD/Asana，最后创建一个 OpenSpec change。
11. 在 `design.md` 写明本次变更与既有规格/历史需求的关系，确认不会无声破坏旧验收标准。
12. 大重构：按 `large-refactor-workflow` 先做 Discovery、RFC、测试基线和 phase 拆分，再询问用户是否启用多 agent。
13. 用户确认启用多 agent 后，先确认 worker 数量、角色、并发数、Claude 是否参与和预计成本，再创建 `.workflow-team/<change-id>/`。
14. 如果项目有 `.codegraph/`，先按 `codegraph-context-guard` 定位入口、调用链和影响面。
15. 如果 PRD / OpenSpec 已确认，且实现涉及多文件、多步骤、复杂测试或回滚，可用 `superpowers:writing-plans` 辅助拆执行计划。
16. 实现前按 `coding-discipline` 确认假设、修改范围和验证标准。
17. 实现时按 `java-coding-standard` 和 `springboot-service-patterns` 写代码。
18. 涉及接口、权限、敏感数据时跑 `springboot-security-review`。
19. 构建失败时用 `java-build-fix`，只做最小修复。
20. 实现后按 `java-test-strategy` 补充单测/集成测试/手动验证。
21. 用 CodeGraph impact 复查影响面，再跑 Java Review、MySQL DB Guard 和 PR Quality Gate。

## Superpowers 怎么用

Superpowers 是可选辅助，不是交付主线。

- 用 `superpowers:brainstorming`：只在需求模糊、新功能探索、方案分歧较大时使用；输出目标、范围、候选方案和待确认问题，供 `prd-writer` 写 PRD。
- 用 `superpowers:writing-plans`：只在 PRD / OpenSpec 已确认，且实现复杂时使用；输出文件清单、步骤、测试和提交节奏，供 OpenSpec `tasks.md` 或实现计划吸收。
- 不用 Superpowers：紧急止血、明确 bugfix、小范围配置、文档调整、单文件小改不强制使用。
- 冲突处理：Superpowers 产物和已确认 PRD / OpenSpec 不一致时，先更新 PRD / OpenSpec，再继续实现。

## 反馈闭环怎么用

用户提交线上问题、重大漏检、回归测试遗漏或 PR 被 revert 时，先按既有流程定位和处理当前问题，再做复盘沉淀。

用户可以这样提交：

```text
问题现象：
严重程度：P0/P1/P2/P3
影响范围：
关联 Asana：
关联 PRD：
关联 OpenSpec change：
关联 PR：
日志 / 截图 / 报错：
期望处理方式：先修复 / 先生成 PRD-OpenSpec / 只调查 / 修复后复盘
```

AI 初步定位后必须先让用户确认：

```text
定位结论：
- 现象：
- 疑似根因：
- 证据：
- 影响范围：
- 修复方案：
- 风险：

请确认：
1. 这个根因是否符合你的业务现场？
2. 你希望现在直接修复，还是先生成 PRD / OpenSpec 后再修？
```

根因明确并处理完成后：

1. 生成 `docs/postmortem/<问题ID>-<问题摘要>.md`。
2. 判断 PRD、OpenSpec、实现、测试、Review、PR Gate、上线观察哪一层漏检。
3. 如果属于可预防问题，把 IMP 记录追加到 `docs/improvements/工作流改进追踪.md`。
4. 标记是否建议升级中央 workflow。
5. 反馈包交给中央 workflow 维护者前必须脱敏。

## MySQL MCP 原则

- 只使用专门的 AI 账号，不使用人工账号。
- AI 账号可以有 `SELECT`、`INSERT`、`UPDATE`、`DELETE`、`CREATE`、`ALTER`，但不给 `DROP`、`TRUNCATE`、`GRANT OPTION`。
- `SELECT`、`SHOW`、`DESCRIBE`、`EXPLAIN` 可直接执行。
- `INSERT`、`UPDATE` 必须先展示 SQL、影响范围和回滚方式。
- `DELETE` 必须先预览影响行，再得到用户明确授权。
- `CREATE`、非破坏性 `ALTER` 必须强确认后才执行。
- `ALTER ... DROP`、`DROP`、`TRUNCATE` 默认拒绝。

## AI 写代码纪律

- 不猜需求；不清楚先补 PRD 的待确认问题，或回 Asana 评论。
- 不做未要求的功能。
- 不做无关重构。
- 每个改动必须能对应 PRD、OpenSpec 或 `tasks.md`。
- 优先小范围、可验证的修改。
- 完成前必须给验证证据。

## CodeGraph 原则

- CodeGraph 是外部 MCP/CLI，不打包进本插件。
- 如果项目存在 `.codegraph/`，陌生代码、跨模块需求、接口/Service/Mapper 改动前先查 CodeGraph。
- 优先查 `context`、`callers`、`callees`、`impact`。
- 修改前查影响面，修改后复查 impact。
- CodeGraph 找不到或信息不足时，再用 `rg` / 文件读取兜底。
- 小改动且文件明确时，可以少查，但最终 Review 仍要能说明影响面。

## 大重构原则

- 大重构不走单个 OpenSpec change。
- 先建 Asana Epic，再写 `docs/refactor/<重构名>/重构RFC.md`。
- 先做 CodeGraph Discovery，再补测试基线。
- 按 phase 拆多个 OpenSpec changes。
- 命中大重构后，不默认启用多 agent；必须先询问用户。
- 用户确认启用后，必须再确认 worker 数量、角色、并发数、Claude 是否参与和预算。
- 启用后创建 `.workflow-team/<change-id>/`，使用 `workflow-config.yaml`、`manifest.md`、`dependencies.md`、`worker-status.md`、`decisions.md`、`merge-log.md` 和任务目录记录状态。
- `concurrency_group` 留空表示可并行；填写名称表示按 `workflow-config.yaml` 的 `max_parallel` 限流，`max_parallel: 1` 才是串行。
- `file_locks` 支持通配符；锁范围重叠时必须串行。当前是文件协议，不是自动锁服务；Leader 需要手动判断，或使用后续辅助脚本检测。
- Claude review 是可选外部能力，需要成员单独安装 `cc-plugin-codex` 并使用 `Claude代码审查任务单.md`。
- 每个 phase 一个小 PR 或多个小 PR。
- 先新增兼容层，再迁移调用方，再切换默认路径，最后删除旧代码。
- 每个 phase 必须可独立 Review、合并、回滚。

## 工具失败处理

工具失败时按 `工具失败处理.md` 执行。不要因为 OpenSpec、CodeGraph、MySQL MCP 或测试命令失败就绕过流程。

## Java / Spring Boot 原则

- 先跟随项目本地 formatter、Checkstyle、Spotless、CI。
- Controller 保持薄，只做 HTTP 层职责。
- Service 负责业务编排、事务、幂等、状态流转。
- Mapper/Repository 只做数据访问。
- 统一异常处理，错误响应不泄露内部异常。
- 参数校验、权限校验、SQL 参数绑定必须在后端完成。
- 构建失败时用最小改动修复，不顺手重构。

## 测试原则

- 行为变更必须有测试，或写明不能自动化的原因。
- Bugfix 必须有回归测试，先复现再修。
- Service 逻辑优先 JUnit 5 + Mockito。
- Controller/API 优先 MockMvc / WebMvcTest。
- Mapper/SQL/事务优先集成测试，不能自动化时必须给 SQL/手动验证证据。
- PR 描述必须列出测试命令和结果。

## 不建议第一版做的事

- 不要全量接入 MCP。
- 不要自动修改 Asana 状态。
- 不要自动开 PR。
- 不要让 AI 账号连接生产写库。
- 不要给 AI 账号 `DROP`、`TRUNCATE`。
- 即使给了 `ALTER`，也必须由 MCP/流程拦截 `ALTER ... DROP` 等破坏性 DDL。
- 不要引入 Go、C++、Swift、Python 等无关流程。
- 不要一开始做自动 multi-agent 编排器；先使用文件协议、Leader 手动调度和用户确认。

先用 3 条真实小需求跑通，再决定要不要自动化。
