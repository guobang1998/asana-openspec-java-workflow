# Asana OpenSpec Java Workflow

这是一个轻量 Codex 插件包，用来把来自 Asana、用户对话、会议纪要、线上问题或技术债的团队需求推进到 PRD、OpenSpec change、Java 实现、MySQL 数据验证、质量门禁和 PR。

## 适用团队

- 需求可能来自 Asana、用户对话、会议纪要、线上问题或技术债想法。
- Asana 用于需求跟踪、排期和状态流转，但不是唯一入口。
- 代码主要是 Java、Spring Boot、MyBatis、SQL。
- 希望用 OpenSpec 记录需求变更，而不是只靠聊天上下文。
- 希望 PR 前有固定的 Java 后端 Review、SQL 性能评审、MySQL 数据安全规则和质量门禁。

## 插件内容

- `prd-writer`：把多入口需求整理成 PRD。
- `setup-workflow`：首次接入仓库时只读发现项目规则，并在确认后写入代码规范地图。
- `wayfinder-workflow`：为跨会话、路径不清的大目标创建本地 map、frontier ticket 和阻塞关系；一次只处理一张票。系统会拒绝遗漏可执行票的 Frontier，Research 必须保留来源并回链票据，全部收敛后生成 `交接.md` 再进入 PRD、OpenSpec 或 RFC。
- `asana-openspec-java-workflow:asana-openspec-delivery`：主交付流程。
- `large-refactor-workflow`：大重构升级流程，按 Epic、RFC、phase/tasks、多 agent 评估、小 PR 串行推进。
- `codegraph-context-guard`：CodeGraph 知识图谱使用规则，防止 AI 乱定位代码。
- `asana-openspec-java-workflow:coding-discipline`：AI 写代码纪律，吸收 Karpathy 的少猜、小改、可验证原则。
- `java-coding-standard`：Java 编码规范，约束命名、分层、异常、日志、MyBatis、测试。
- `springboot-service-patterns`：Spring Boot 服务设计模式，约束 Controller/Service/Mapper、事务、分页、缓存、异步。
- `springboot-security-review`：Spring Boot 安全评审，检查认证、授权、输入校验、SQL 注入、密钥、敏感日志。
- `java-build-fix`：Java/Maven/Gradle 构建失败修复流程。
- `java-test-strategy`：Java 测试策略，规定单测、集成测试、手动验证和 PR 测试说明。
- `java-backend-review`：Java/Spring/MyBatis 后端专项评审。
- `asana-openspec-java-workflow:sql-performance-review`：SQL 编码规则与接口性能审查；写 SQL 时先遵循，审查代码时再检查 MyBatis SQL、N+1、分页、排序、索引和 `EXPLAIN` 证据。
- `mysql-db-guard`：MySQL MCP / AI 专用账号的读写安全规则。
- `pr-quality-gate`：PR 前质量门禁。
- `docs/workflow/大需求分段审核与高风险复查协议.md`：大需求 phase 分段审核规则，高风险复查只在风险命中时触发。
- Superpowers 外部辅助能力：`superpowers:brainstorming` 用于模糊需求和新功能探索澄清，`superpowers:writing-plans` 用于复杂实现拆计划；不替代 PRD / OpenSpec 主流程。
- `assets/templates/`：PRD、Asana 字段、OpenSpec、MySQL、PR Review、AGENTS 模板、多 Agent 任务单、Claude 审查任务单和结构化状态模板。

## 多入口需求路由

需求入口不限于 Asana。AI 先判断入口类型，再决定下一步。

优先级：`setup-workflow > wayfinder-workflow > superpowers:brainstorming`。

<!-- route-priority: setup > wayfinder > brainstorming -->

| 入口类型 | 第一动作 |
|---|---|
| 新接入 Java 仓库，或缺少 `docs/agents/代码规范地图.md` | `setup-workflow` 只读发现；写地图前等待确认 |
| 跨会话大目标、路径不清、多个决策或阻塞关系未清 | `wayfinder-workflow` 只画 map，不直接实现 |
| 明确 Asana 需求 | 读取 Asana，生成或完善 PRD |
| 用户对话中的明确需求 | 生成 PRD 草稿，标记 `需求来源：用户对话` |
| 单会话、小范围且目标基本可判断的模糊需求 | 先做轻量澄清；范围不清或方案分歧较大时进入 `superpowers:brainstorming` |
| 口头/会议需求 | 先生成需求 Intake 或 PRD 草稿，列待确认问题 |
| 新功能探索 | 跨会话或路径不清时先 Wayfinder；否则用 `superpowers:brainstorming` |
| 技术债/重构想法 | Destination、范围或首批决策不清时先 Wayfinder；清楚后再 Discovery / RFC |
| 线上问题 | 先定位根因，再决定修复、PRD/OpenSpec 或只调查 |

没有 Asana 不阻塞启动，但需要团队协作、排期或跨团队跟踪时，应后补 Asana。
没有确认 PRD / OpenSpec，不进入行为变更实现。

轻量澄清适用于小范围、低风险、目标基本可判断的模糊需求，只要求输出目标、范围、不做什么、待确认问题，并进入 PRD 草稿。
正式 `superpowers:brainstorming` 只适用于未命中 setup 或 Wayfinder 的新功能探索、方案分歧较大、用户体验/业务口径不清、范围不清、可能演化成重构的需求。
轻量澄清和正式 `superpowers:brainstorming` 都不能直接进入实现；二者产物都只能作为 PRD 输入。

## 推荐落地方式

1. 先按 `团队安装指南.md` 把本工具包 clone 到 Codex plugins 目录。
2. 再按 `安装配置指南.md` 安装和启用 OpenSpec、CodeGraph、Asana/GitHub Connector、MySQL MCP 等外部工具。
3. 先看 `流程图.md`，确认团队理解“小需求”和“大重构”的分流方式。
4. 如需团队跟踪，在 Asana 建需求项目和字段，参考 `assets/templates/Asana字段模板.md`。
5. 在 Java 项目仓库初始化 OpenSpec。
6. 把 `assets/templates/AGENTS模板.md` 内容合并到项目 `AGENTS.md`。
7. 为 MySQL MCP 准备一个专用 AI 账号，参考 `assets/templates/MySQL数据库安全模板.md`。
8. 每条需求先判断是否缺代码规范地图；缺失时先运行 `setup-workflow` 只读发现。
9. 再判断是否跨会话且路径不清；命中时先用 `wayfinder-workflow` 画 map，一次只推进一张 frontier ticket。通常不需要单独 `@` 它，主插件会按入口优先级自动路由；需要强制建图或继续指定 map 时再显式调用。
10. 未命中 Wayfinder 的模糊需求、新功能探索或方案分歧较大，再做轻量澄清或 `superpowers:brainstorming`；输出只作为 PRD 输入。
11. 小需求：先产出 PRD，再检查既有 `openspec/specs/*`、active changes、历史 PRD/Asana/会议纪要/问题记录，最后创建一个 OpenSpec change。
12. 在 `design.md` 写明本次变更与既有规格/历史需求的关系，确认不会无声破坏旧验收标准。
13. 大重构：按 `large-refactor-workflow` 先做 Discovery、RFC、测试基线和 phase 拆分，再询问用户是否启用多 agent。
14. 大重构每个 phase 开始前写实现前边界声明，phase 出口填写分段审核基础判断。
15. 命中 A 类或明显 B 类风险时，按 `docs/workflow/大需求分段审核与高风险复查协议.md` 判断是否触发高风险复查。
16. 用户确认启用多 agent 后，先确认 worker 数量、角色、并发数、Claude 是否参与和预计成本，再创建 `.workflow-team/<change-id>/`。
17. 如果项目有 `.codegraph/`，先按 `codegraph-context-guard` 定位入口、调用链和影响面。
18. 如果 PRD / OpenSpec 已确认，且实现涉及多文件、多步骤、复杂测试或回滚，可用 `superpowers:writing-plans` 辅助拆执行计划。
19. 实现前优先按 `asana-openspec-java-workflow:coding-discipline` 确认假设、修改范围和验证标准；如果当前会话没有暴露该 skill，按项目 `AGENTS.md` 的“AI 写代码纪律”执行同一检查。
20. 实现时按 `java-coding-standard` 和 `springboot-service-patterns` 写代码；涉及 Mapper/SQL、列表、搜索、统计、导出、分页或排序时，先按 `sql-performance-review` 编码模式设计和实现。
21. 涉及接口、权限、敏感数据时跑 `springboot-security-review`。
22. 构建失败时用 `java-build-fix`，只做最小修复。
23. 实现后按 `java-test-strategy` 补充单测/集成测试/手动验证。
24. 用 CodeGraph impact 复查影响面，填写 `PR评审清单.md` 的分段审核基础判断，再跑 Java Review、SQL Performance Review、MySQL DB Guard 和 PR Quality Gate。

## 常用提示词

多入口需求：

```text
请按 asana-openspec-java-workflow 自动处理这个目标：先加载仓库代码规范地图；缺失时先只读 setup。若目标跨会话且路径不清，先创建 Wayfinder map；否则按 PRD、OpenSpec 和 Java 后端交付流程推进。不要直接写代码。
```

模糊需求：

```text
请按 asana-openspec-java-workflow 自动处理这个目标：先检查是否需要 setup；若跨会话且路径不清，先创建 Wayfinder map；未命中时再用 brainstorming 澄清，不要写代码。
```

会议/口头需求：

```text
这不是 Asana 任务，是会议里提到的想法。请先按入口优先级自动分流：缺代码规范地图先 setup；跨会话且路径不清先建 Wayfinder map；未命中时再生成 PRD 草稿并列待确认问题。
```

SQL 性能专项审查。这个提示词只审 SQL/Mapper/查询接口，不自动展开 PRD、OpenSpec、完整 Java Review 或 PR Gate：

```text
请使用本插件的 asana-openspec-java-workflow:sql-performance-review 审查这次 Mapper/SQL/列表接口改动；如果当前可用清单显示为 Asana OpenSpec Java Workflow:sql-performance-review 或只暴露短名 sql-performance-review，请按当前清单名称调用。只审 SQL/Mapper/查询接口，重点看分页、排序、N+1、索引匹配和 EXPLAIN 证据。
```

## Superpowers 怎么用

Superpowers 是可选辅助，不是交付主线。

- 用轻量澄清：小范围、低风险、目标基本可判断的模糊需求，可以先输出目标、范围、不做什么、待确认问题，供 `prd-writer` 写 PRD 草稿。
- 用 `superpowers:brainstorming`：仅在未命中 setup 或 Wayfinder，且需求模糊、新功能探索、方案分歧较大、用户体验/业务口径不清、范围不清或可能演化成重构时使用；输出目标、范围、候选方案和待确认问题，供 `prd-writer` 写 PRD。
- 轻量澄清和正式 `superpowers:brainstorming` 都不能直接进入实现；产物都只能作为 PRD 输入。
- 用 `superpowers:writing-plans`：只在 PRD / OpenSpec 已确认，且实现复杂时使用；输出文件清单、步骤、测试和提交节奏，供 OpenSpec `tasks.md` 或实现计划吸收。
- 不用 Superpowers：紧急止血、明确 bugfix、小范围配置、文档调整、单文件小改不强制使用。
- 冲突处理：Superpowers 产物和已确认 PRD / OpenSpec 不一致时，先更新 PRD / OpenSpec，再继续实现。

## 反馈闭环怎么用

用户提交线上问题、重大漏检、回归测试遗漏或 PR 被 revert 时，先按既有流程定位和处理当前问题，再做复盘沉淀。

用户可以这样提交：

```text
问题现象：
严重程度：P0/P1/P2/P3
影响范围：
关联 Asana：
关联 PRD：
关联 OpenSpec change：
关联 PR：
日志 / 截图 / 报错：
期望处理方式：先修复 / 先生成 PRD-OpenSpec / 只调查 / 修复后复盘
```

AI 初步定位后必须先让用户确认：

```text
定位结论：
- 现象：
- 疑似根因：
- 证据：
- 影响范围：
- 修复方案：
- 风险：

请确认：
1. 这个根因是否符合你的业务现场？
2. 你希望现在直接修复，还是先生成 PRD / OpenSpec 后再修？
```

根因明确并处理完成后：

1. 生成 `docs/postmortem/<问题ID>-<问题摘要>.md`。
2. 判断 PRD、OpenSpec、实现、测试、Review、PR Gate、上线观察哪一层漏检。
3. 如果属于可预防问题，把 IMP 记录追加到 `docs/improvements/工作流改进追踪.md`。
4. 标记是否建议升级中央 workflow。
5. 反馈包交给中央 workflow 维护者前必须脱敏。

## MySQL MCP 原则

- 只使用专门的 AI 账号，不使用人工账号。
- AI 账号可以有 `SELECT`、`INSERT`、`UPDATE`、`DELETE`、`CREATE`、`ALTER`，但不给 `DROP`、`TRUNCATE`、`GRANT OPTION`。
- `SELECT`、`SHOW`、`DESCRIBE`、`EXPLAIN` 可直接执行。
- `INSERT`、`UPDATE` 必须先展示 SQL、影响范围和回滚方式。
- `DELETE` 必须先预览影响行，再得到用户明确授权。
- `CREATE`、非破坏性 `ALTER` 必须强确认后才执行。
- `ALTER ... DROP`、`DROP`、`TRUNCATE` 默认拒绝。

## SQL 性能评审原则

- 涉及 Mapper/SQL、列表、搜索、统计、导出、分页或排序时，写代码必须遵循 `sql-performance-review` 编码模式，审查代码必须使用 `sql-performance-review` 审查模式。
- 列表/搜索接口必须有默认分页、最大 pageSize、稳定排序和排序字段白名单。
- Service 不能在循环里按行查库造成 N+1；优先批量查询或 read model。
- 关键 SQL 必须有 `EXPLAIN` 证据，或写明小表/低频/低风险理由。
- 高风险 SQL 没有 `EXPLAIN`，PR Gate 不能 PASS。
- 缓存不能替代 SQL 根因修复；先看索引、查询形状、返回字段和分页策略。

## AI 写代码纪律

`coding-discipline` 是本插件随包提供的 skill。在 Codex 可用 skills 清单里通常显示为 `asana-openspec-java-workflow:coding-discipline`。若 Codex 当前会话提示没有该 skill，不阻塞试跑，直接按本节和项目 `AGENTS.md` 执行。

- 不猜需求；不清楚先补 PRD 的待确认问题，有 Asana 时可回 Asana 评论。
- 不做未要求的功能。
- 不做无关重构。
- 每个改动必须能对应 PRD、OpenSpec 或 `tasks.md`。
- 优先小范围、可验证的修改。
- 完成前必须给验证证据。

## CodeGraph 原则

- CodeGraph 是外部 MCP/CLI，不打包进本插件。
- 如果项目存在 `.codegraph/`，陌生代码、跨模块需求、接口/Service/Mapper 改动前先查 CodeGraph。
- 优先查 `context`、`callers`、`callees`、`impact`。
- 修改前查影响面，修改后复查 impact。
- CodeGraph 找不到或信息不足时，再用 `rg` / 文件读取兜底。
- 小改动且文件明确时，可以少查，但最终 Review 仍要能说明影响面。

## 大重构原则

- 大重构不走单个 OpenSpec change。
- 有团队排期或跨团队跟踪时先建或关联 Asana Epic，再写 `docs/refactor/<重构名>/重构RFC.md`；没有 Asana 时先写 RFC 草稿并记录需求来源。
- 先做 CodeGraph Discovery，再补测试基线。
- 按 phase 拆多个 OpenSpec changes。
- 每个 phase 开工前写实现前边界声明，phase 出口填写分段审核基础判断。
- 高风险复查按风险触发，不按工具触发；A 类或明显 B 类风险必须说明是否触发以及跳过依据。
- 命中大重构后，不默认启用多 agent；必须先询问用户。
- 用户确认启用后，必须再确认 worker 数量、角色、并发数、Claude 是否参与和预算。
- 启用后创建 `.workflow-team/<change-id>/`，使用 `workflow-config.yaml`、`manifest.md`、`dependencies.md`、`worker-status.md`、`decisions.md`、`merge-log.md` 和任务目录记录状态。
- `concurrency_group` 留空表示可并行；填写名称表示按 `workflow-config.yaml` 的 `max_parallel` 限流，`max_parallel: 1` 才是串行。
- `file_locks` 支持通配符；锁范围重叠时必须串行。当前是文件协议，不是自动锁服务；Leader 需要手动判断，或使用后续辅助脚本检测。
- Claude review 是可选外部能力，需要成员单独安装 `cc-plugin-codex` 并使用 `Claude代码审查任务单.md`。
- 每个 phase 一个小 PR 或多个小 PR。
- 先新增兼容层，再迁移调用方，再切换默认路径，最后删除旧代码。
- 每个 phase 必须可独立 Review、合并、回滚。

## 工具失败处理

工具失败时按 `工具失败处理.md` 执行。不要因为 OpenSpec、CodeGraph、MySQL MCP 或测试命令失败就绕过流程。

## Java / Spring Boot 原则

- 先跟随项目本地 formatter、Checkstyle、Spotless、CI。
- Controller 保持薄，只做 HTTP 层职责。
- Service 负责业务编排、事务、幂等、状态流转。
- Mapper/Repository 只做数据访问。
- 统一异常处理，错误响应不泄露内部异常。
- 参数校验、权限校验、SQL 参数绑定必须在后端完成。
- 构建失败时用最小改动修复，不顺手重构。

## 测试原则

- 行为变更必须有测试，或写明不能自动化的原因。
- Bugfix 必须有回归测试，先复现再修。
- Service 逻辑优先 JUnit 5 + Mockito。
- Controller/API 优先 MockMvc / WebMvcTest。
- Mapper/SQL/事务优先集成测试，不能自动化时必须给 SQL/手动验证证据；关键查询补 `EXPLAIN` 或等价执行计划证据。
- PR 描述必须列出测试命令和结果。

## 不建议第一版做的事

- 不要全量接入 MCP。
- 不要自动修改 Asana 状态。
- 不要自动开 PR。
- 不要让 AI 账号连接生产写库。
- 不要给 AI 账号 `DROP`、`TRUNCATE`。
- 即使给了 `ALTER`，也必须由 MCP/流程拦截 `ALTER ... DROP` 等破坏性 DDL。
- 不要引入 Go、C++、Swift、Python 等无关流程。
- 不要一开始做自动 multi-agent 编排器；先使用文件协议、Leader 手动调度和用户确认。

先用 3 条真实小需求跑通，再决定要不要自动化。
