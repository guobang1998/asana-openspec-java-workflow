# Asana OpenSpec Java Workflow

这是一个轻量 Codex 插件包，用来把团队需求从 Asana 推进到 PRD、OpenSpec change、Java 实现、MySQL 数据验证、质量门禁和 PR。

## 适用团队

- 需求入口在 Asana。
- 代码主要是 Java、Spring Boot、MyBatis、SQL。
- 希望用 OpenSpec 记录需求变更，而不是只靠聊天上下文。
- 希望 PR 前有固定的 Java 后端 Review、MySQL 数据安全规则和质量门禁。

## 插件内容

- `prd-writer`：把 Asana 需求整理成 PRD。
- `asana-openspec-delivery`：主交付流程。
- `large-refactor-workflow`：大重构升级流程，按 Epic、RFC、多阶段 OpenSpec changes、小 PR 串行推进。
- `codegraph-context-guard`：CodeGraph 知识图谱使用规则，防止 AI 乱定位代码。
- `coding-discipline`：AI 写代码纪律，吸收 Karpathy 的少猜、小改、可验证原则。
- `java-coding-standard`：Java 编码规范，约束命名、分层、异常、日志、MyBatis、测试。
- `springboot-service-patterns`：Spring Boot 服务设计模式，约束 Controller/Service/Mapper、事务、分页、缓存、异步。
- `springboot-security-review`：Spring Boot 安全评审，检查认证、授权、输入校验、SQL 注入、密钥、敏感日志。
- `java-build-fix`：Java/Maven/Gradle 构建失败修复流程。
- `java-test-strategy`：Java 测试策略，规定单测、集成测试、手动验证和 PR 测试说明。
- `java-backend-review`：Java/Spring/MyBatis 后端专项评审。
- `mysql-db-guard`：MySQL MCP / AI 专用账号的读写安全规则。
- `pr-quality-gate`：PR 前质量门禁。
- `assets/templates/`：PRD、Asana 字段、OpenSpec、MySQL、PR Review、AGENTS 模板。

## 推荐落地方式

0. 先按 `团队安装指南.md` 把本工具包 clone 到 Codex plugins 目录。
1. 再按 `安装配置指南.md` 安装和启用 OpenSpec、CodeGraph、Asana/GitHub Connector、MySQL MCP 等外部工具。
1. 先看 `流程图.md`，确认团队理解“小需求”和“大重构”的分流方式。
2. 在 Asana 建需求项目和字段，参考 `assets/templates/Asana字段模板.md`。
3. 在 Java 项目仓库初始化 OpenSpec。
4. 把 `assets/templates/AGENTS模板.md` 内容合并到项目 `AGENTS.md`。
5. 为 MySQL MCP 准备一个专用 AI 账号，参考 `assets/templates/MySQL数据库安全模板.md`。
6. 每条需求先判断是小需求还是大重构。
7. 小需求：先产出 PRD，再创建一个 OpenSpec change。
8. 大重构：按 `large-refactor-workflow` 先做 Discovery、RFC、测试基线和 phase 拆分，再为每个 phase 创建 OpenSpec change。
9. 如果项目有 `.codegraph/`，先按 `codegraph-context-guard` 定位入口、调用链和影响面。
10. 实现前按 `coding-discipline` 确认假设、修改范围和验证标准。
11. 实现时按 `java-coding-standard` 和 `springboot-service-patterns` 写代码。
12. 涉及接口、权限、敏感数据时跑 `springboot-security-review`。
13. 构建失败时用 `java-build-fix`，只做最小修复。
14. 实现后按 `java-test-strategy` 补充单测/集成测试/手动验证。
15. 用 CodeGraph impact 复查影响面，再跑 Java Review、MySQL DB Guard 和 PR Quality Gate。

## MySQL MCP 原则

- 只使用专门的 AI 账号，不使用人工账号。
- AI 账号可以有 `SELECT`、`INSERT`、`UPDATE`、`DELETE`、`CREATE`、`ALTER`，但不给 `DROP`、`TRUNCATE`、`GRANT OPTION`。
- `SELECT`、`SHOW`、`DESCRIBE`、`EXPLAIN` 可直接执行。
- `INSERT`、`UPDATE` 必须先展示 SQL、影响范围和回滚方式。
- `DELETE` 必须先预览影响行，再得到用户明确授权。
- `CREATE`、非破坏性 `ALTER` 必须强确认后才执行。
- `ALTER ... DROP`、`DROP`、`TRUNCATE` 默认拒绝。

## AI 写代码纪律

- 不猜需求；不清楚先补 PRD 的待确认问题，或回 Asana 评论。
- 不做未要求的功能。
- 不做无关重构。
- 每个改动必须能对应 PRD、OpenSpec 或 `tasks.md`。
- 优先小范围、可验证的修改。
- 完成前必须给验证证据。

## CodeGraph 原则

- CodeGraph 是外部 MCP/CLI，不打包进本插件。
- 如果项目存在 `.codegraph/`，陌生代码、跨模块需求、接口/Service/Mapper 改动前先查 CodeGraph。
- 优先查 `context`、`callers`、`callees`、`impact`。
- 修改前查影响面，修改后复查 impact。
- CodeGraph 找不到或信息不足时，再用 `rg` / 文件读取兜底。
- 小改动且文件明确时，可以少查，但最终 Review 仍要能说明影响面。

## 大重构原则

- 大重构不走单个 OpenSpec change。
- 先建 Asana Epic，再写 `docs/refactor/<重构名>/重构RFC.md`。
- 先做 CodeGraph Discovery，再补测试基线。
- 按 phase 拆多个 OpenSpec changes。
- 每个 phase 一个小 PR 或多个小 PR。
- 先新增兼容层，再迁移调用方，再切换默认路径，最后删除旧代码。
- 每个 phase 必须可独立 Review、合并、回滚。

## 工具失败处理

工具失败时按 `工具失败处理.md` 执行。不要因为 OpenSpec、CodeGraph、MySQL MCP 或测试命令失败就绕过流程。

## Java / Spring Boot 原则

- 先跟随项目本地 formatter、Checkstyle、Spotless、CI。
- Controller 保持薄，只做 HTTP 层职责。
- Service 负责业务编排、事务、幂等、状态流转。
- Mapper/Repository 只做数据访问。
- 统一异常处理，错误响应不泄露内部异常。
- 参数校验、权限校验、SQL 参数绑定必须在后端完成。
- 构建失败时用最小改动修复，不顺手重构。

## 测试原则

- 行为变更必须有测试，或写明不能自动化的原因。
- Bugfix 必须有回归测试，先复现再修。
- Service 逻辑优先 JUnit 5 + Mockito。
- Controller/API 优先 MockMvc / WebMvcTest。
- Mapper/SQL/事务优先集成测试，不能自动化时必须给 SQL/手动验证证据。
- PR 描述必须列出测试命令和结果。

## 不建议第一版做的事

- 不要全量接入 MCP。
- 不要自动修改 Asana 状态。
- 不要自动开 PR。
- 不要让 AI 账号连接生产写库。
- 不要给 AI 账号 `DROP`、`TRUNCATE`。
- 即使给了 `ALTER`，也必须由 MCP/流程拦截 `ALTER ... DROP` 等破坏性 DDL。
- 不要引入 Go、C++、Swift、Python 等无关流程。
- 不要做复杂 multi-agent 编排。

先用 3 条真实小需求跑通，再决定要不要自动化。
